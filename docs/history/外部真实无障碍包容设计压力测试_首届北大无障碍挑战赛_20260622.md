## 一、为什么选这个样本

## 二、暴露出的协议缺口

旧协议中已有“可及性、公平、弱势群体保护”等字段，但分散在社区治理、参与者保护、赛事承办、教学和就业等 trace 里，缺少一个专门处理“能否独立、安全、有尊严地完成真实任务”的无障碍主 trace。

如果没有这一层，AI 很容易写出这些不可靠结论：

1. “有坡道，所以轮椅使用者可达。”
2. “有适老化模式，所以老年人可用。”
3. “有字幕/语音播报，所以信息可及。”
4. “有无障碍卫生间和标识，所以场馆无障碍。”
5. “有志愿者陪同，所以残障者可以平等参与。”
6. “获奖方案关注残障群体，所以已经实现包容设计。”

这些判断的共同问题是把设施、功能、善意或奖项当成真实使用证据。真正需要追踪的是目标用户、真实任务、全路径可达、辅助技术兼容、独立完成率、错误率、求助依赖、失败任务、成本负担、安全尊严、隐私、维护和投诉反馈。

## 五、判断规则

1. 单个设施、功能、标识、志愿者陪同、用户好评、验收证书和获奖方案，只能证明“有无障碍相关设计或材料”，不能直接证明真实可用。
2. “符合规范”不等于“完整任务可完成”；规范底线、全路径可达、辅助技术兼容和真实用户使用是不同证据。
3. 志愿者陪同可能是必要支持，但不能替代独立使用、尊严保护和非隔离式参与；如果必须依赖人工帮助，必须说明帮助边界和可持续性。
4. 无障碍测试必须区分不同障碍类型、年龄、语言、识字水平、神经多样性和交叉弱势；少量体验者不能代表全部目标人群。
5. 涉及未成年人、弱势群体、服务对象或可识别个人经历时同步看 `stakeholder_consent_participant_protection_trace`；涉及 App、网站、平台算法和个人信息时同步看 `data_algorithm_privacy_governance_trace`；涉及场馆赛事或疏散时同步看 `event_hosting_operation_trace` 和 `emergency_safety_risk_response_trace`。

## 六、迁移到真实项目的用法

| 项目类型 | 常见误判 | 最小补证 |
| --- | --- | --- |
| 无障碍公益创新赛 | 方案获奖等于真实可用 | 共创记录、真实用户任务测试、失败任务和改进闭环 |
| 校园/场馆无障碍 | 有坡道/电梯/厕所等于全路径可达 | 入口到目的地全路径、厕所、座席、导向、疏散和维护 |
| 网站/App 适老化 | 大字模式和语音播报等于可用 | 屏幕阅读器、键盘操作、焦点、对比、任务完成和错误率 |
| 无障碍阅读 | 有音频或电子文本等于信息可及 | 格式、版权授权、辅助技术兼容、易读版本和读者反馈 |
| 公共服务窗口 | 有志愿者等于平等服务 | 独立办理、隐私保护、排队等待、投诉反馈和人员培训 |
| 特殊教育支持 | 有课程或设备等于学习机会平等 | 个别化需求、教师支持、学习任务完成和长期维护 |

## 九、最小结论门

没有以下材料之一时，不得写“独立安全使用、平等参与、信息可及、体验友好、尊严被保护、可规模复制”：

1. 明确目标人群和真实使用场景。
2. 适用法律、标准、指南和项目要求。
3. 与目标用户共同设计或需求访谈记录。
4. 关键任务全路径完成测试。
5. 独立完成率、错误率、用时和求助依赖。
6. 辅助技术兼容性和设备可获得性。
7. 失败任务、障碍清单和修复记录。
8. 安全、尊严、隐私、非污名化和非隔离式参与说明。
9. 维护运营、人员培训、反馈投诉和长期责任。

证据不足时使用替代表达：

1. “已形成无障碍设计或改造材料。”
2. “目前只能证明关注了可及性问题。”
3. “真实用户可用性和长期维护仍需补充证据。”
4. “该结论不能外推为残障者/老年人可独立安全使用或平等参与。”
