# 外部真实志愿服务压力测试：2026 大学生志愿服务西部计划

current_as_of: 2026-06-22

## 1. 本轮选择

本轮继续执行用户修正后的范围：真实项目不只包括 AI 科研，而应覆盖现实时间中的比赛项目、长期组织项目、公共服务项目和真实交付项目。

选择样本：2026 年大学生志愿服务西部计划。

样本理由：

1. 它不是一次性活动，而是 1-3 年长期基层志愿服务项目。
2. 它同时包含公开招募、组织选拔、集中派遣、岗位服务、权益保障、考核鉴定和离岗交接。
3. 它容易被写成“青年热情、奔赴西部、无私奉献”的故事，但真实项目治理必须证明岗位匹配、基层需求、服务质量、志愿者权益和安全健康。
4. 它能迫使协议区分“志愿故事”和“志愿服务运行成效”。

## 3. 新失败模式

```yaml
failure_mode:
  id: F63-01
  name: volunteer_story_impact_gap
  symptom: 把招募人数、报名热情、服务时长、宣传报道、志愿者故事、表彰证书、活动照片、地方感谢或满意度直接写成基层需求被解决、公共服务改善、志愿者成长、组织运行有效、权益保障到位或项目可持续
  root_cause: 把公益叙事当作服务运行证据，没有区分招募、选拔、派遣、岗位匹配、在岗服务、权益保障、安全健康、考核鉴定和离岗交接
  trigger_signal:
    - 用户要求写西部计划、研支团、赛会志愿者、基层志愿服务、公益服务队、社区志愿服务或志愿服务项目大赛
    - 用户希望证明社会影响、基层服务成效、志愿者成长、长期机制或组织能力
    - 材料中主要证据是招募通知、宣传稿、人物故事、服务时长、证书、合影、感谢信或满意度
  corrected_behavior: 启用 volunteer_service_operation_trace，先核验真实来源和项目阶段，再追踪服务对象和地方需求、岗位职责、志愿者资格与健康条件、协议权责、服务单位责任、培训考核、补贴保险住宿、风险处置、意见反馈、服务鉴定、离岗交接和结果证据；证据不足时降级为招募派遣机制、在岗服务过程材料或待验证的服务成效假设。
  detection_check: 最终文本中是否把“报名/入选/派遣/服务/考核/交接/结果”分开，是否说明哪些主张只能由协议或政策支持、哪些必须由岗位现场和服务对象反馈支持。
```

## 5. 可落地的项目写法变化

旧写法：

> 2026 年西部计划预计招募约 6 万名志愿者，广大青年奔赴西部基层，以青春力量服务乡村振兴，充分体现了志愿服务的社会价值。

修正后写法：

> 2026 年西部计划具备较完整的长期志愿服务运行机制：官方政策问答明确招募时间、服务期和 8 个专项，协议模板进一步把岗位、培训、补贴保险、住宿工作条件、安全健康、考核鉴定和离岗交接写入责任链。这个证据足以说明项目制度框架较完整，但还不能直接证明某个县域公共服务已改善。若要证明“服务有效”，还需要进入具体岗位，收集地方需求来源、服务日志、服务对象反馈、负责人评价、风险事件处理、考核鉴定和交接记录。
