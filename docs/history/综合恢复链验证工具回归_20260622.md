# 综合恢复链验证工具回归

current_as_of：2026-06-22

## 1. 小题背景

第 161/162 节机械 EOF 追加工具收口后，主代理又用一段临时 Python 脚本做综合恢复链验证。第一次脚本把 `^\s+##` 当作缩进伪标题检查；由于 `\s` 会匹配换行，正常二级标题被误判为伪标题，造成一次 false positive。

这说明恢复链收口不能长期依赖每轮临时手写脚本：检查口径会漂移，正则细节会出错，发布包、变更清单、机器看版和历史旧入口之间也容易漏扫。

## 2. 暴露的失败模式

```yaml
failure_mode:
  failure_id: ad_hoc_recovery_scan_drift
  symptom: 临时扫描脚本误判伪标题，或每轮只用零散 rg 检查恢复入口
  root_cause: 综合恢复链验证没有沉淀为可复用工具，检查范围和正则口径由当轮上下文临时决定
  trigger_signal:
    - post-change scan 需要同时检查 long_log、release_package、change_inventory、route_table 和 protocol
    - 脚本中出现新写正则、历史旧入口过滤、pending 过滤或 QA 降级判断
  corrected_behavior:
    - 使用 tools/verify_governance_recovery.py 或等价综合扫描
    - 显式传入 latest_heading 和 expected_route
    - 同时校验 EOF 最新看版、发布包、变更清单、历史旧入口标签和 QA 降级状态
  detection_check:
    - 工具输出 failure_count 为 0
    - latest_pending_count 为 0
    - old_current_markers_historical 为 true
    - render_blocked_preserved 为 true
```
