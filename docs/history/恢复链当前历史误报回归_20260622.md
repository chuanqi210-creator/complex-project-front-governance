## 1. 小题触发

上一轮最终扫描中，`rg` 同时命中了：

1. 当前恢复入口：`## 152. 当前机器看版` 与 `continue_self_optimization_with_next_high_leverage_real_project_regression`。
2. 历史记录：第 150 节恢复入口和 `run_dynamic_stage_controller_regressions_on_next_real_project_without_adding_specific_trace_directory`。

## 3. 暴露的共性失败模式

failure_mode_id: `current_historical_scope_confusion`

症状：发布包、变更清单和长日志中会同时存在“当前恢复入口”和“历史恢复记录”。如果扫描命令只做全文关键词匹配，旧章节、历史表格或失败案例中的旧 route 会被误判成当前 stale，导致 AI 修错地方或删除合法历史证据。

可迁移场景：

1. changelog 里保留旧版本状态。
2. 长日志保留旧机器看版。
3. 发布包保留历史失败模式和修复记录。
4. QA 记录保留曾经 blocked / stale 的旧状态。

## 4. 写回规则

```yaml
current_scope_recovery_scan:
  current_scope:
    - 发布包“恢复入口”段
    - 变更清单“当前恢复入口”段
    - 长日志 EOF 最新机器看版
  historical_record_scope:
    - 标注为“历史记录”的表格行
    - 旧章节机器看版
    - 失败模式案例中的旧 route
  decision_rule:
    - current_scope 中出现旧入口才算 stale
    - historical_record_scope 中出现旧入口只算 archive_hit
    - 宽泛 rg 只能作为候选发现，不能单独作为 stale 证据
```

最小结论：恢复链扫描必须先定 scope，再判定 stale；不能用全文命中替代当前状态解析。
