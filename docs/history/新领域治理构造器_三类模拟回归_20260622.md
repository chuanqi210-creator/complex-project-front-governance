# 新领域治理构造器三类模拟回归 2026-06-22

用途：验证 `new_domain_governance_builder` 在不新增具体领域 trace 目录的情况下，能对陌生工程/软件、社会实践/公共服务、开源软件项目形成可执行治理镜头。

## R86-01 陌生工程/软件/产品项目

模拟项目：团队做了一个低成本校园仓储搬运机器人，有 demo 视频和一次现场演示，希望写成“可在校园仓储场景部署”。

```yaml
new_domain_governance_builder:
  domain_hypothesis: 工程原型可能有真实应用价值，但 demo 只能证明局部可行
  primary_claim_to_prove: 校园仓储搬运机器人是否具备真实部署前的工程验证基础
  category_route:
    primary_library: engineering_software_product
    high_risk_regulatory_tags:
      - safety
      - intellectual_property
  selected_lenses:
    - claim_ladder
    - execution_validation
    - risk_ethics_permission
    - operation_handoff
    - transfer_boundary
  domain_specific_questions:
    - question: demo 的场地、路线、负载、时长、失败次数和人工接管条件是什么？
      why_it_matters: 防止把一次演示外推成长期可靠
      evidence_needed: 测试记录、失败样例、视频元数据、场地和负载说明
    - question: 部署场景中谁负责安全、维护、充电、急停和事故处理？
      why_it_matters: 工程可部署必须有运行责任
      evidence_needed: 安全预案、维护计划、责任人、验收标准
  micro_task_to_execute: 拆出 demo 到部署之间缺失的 5 项证据链
  actual_execution_evidence:
    - demo 只能证明机器人在一个受控场景中完成一次任务
    - 真实部署至少需要重复测试、失败记录、安全机制、维护责任和用户验收
  observed_result: 可以写“形成可测试工程原型”，不能写“已可校园部署”
  downgrade_rule: 缺少连续运行、故障样例和安全验收时，降级为竞赛/课程条件下的工程原型
  promotion_decision: use_once
```

## R86-02 陌生社会实践/公共服务项目

模拟项目：支队在县域学校做了 AI 学习工作坊，学生反馈很好，希望写成“显著提升县中学生 AI 素养和未来规划能力”。

```yaml
new_domain_governance_builder:
  domain_hypothesis: 工作坊可能带来认知触发，但短期满意度不能证明长期能力提升
  primary_claim_to_prove: AI 学习工作坊是否对学生认知、技能或路径感产生可观察改变
  category_route:
    primary_library: education_practice_public_service
    high_risk_regulatory_tags:
      - minors
      - privacy
  selected_lenses:
    - claim_ladder
    - stakeholder_context
    - execution_validation
    - risk_ethics_permission
    - deliverable_storyline
  domain_specific_questions:
    - question: 学生在课前、课后分别能完成什么具体任务？
      why_it_matters: 防止把喜欢课堂外推为学会或能力提升
      evidence_needed: 课前小测、课后作品、学生反思、分层样本
    - question: 访谈、照片、故事和作品是否获得学生与监护/学校授权？
      why_it_matters: 未成年人项目不能用公益善意替代授权
      evidence_needed: 最小采集方案、匿名化规则、公开授权边界
  micro_task_to_execute: 写出一条最终报告主张到课堂证据字段的追踪链
  actual_execution_evidence:
    - “学生反馈好”只能支撑接受度
    - “学生能用 AI 完成一个自选学习任务并解释局限”才支撑初步学习结果
  observed_result: 可以写“形成短期认知触发和作品证据”，不能写“显著提升长期 AI 素养”
  downgrade_rule: 缺少基线、作品、后测或追踪时，降级为短期工作坊观察和待验证假设
  promotion_decision: use_once
```

## R86-03 陌生软件/开源项目

模拟项目：学生给一个开源数据可视化库提交了 bugfix PR，本地测试通过，希望写成“解决了上游项目关键问题并被社区采纳”。

```yaml
new_domain_governance_builder:
  domain_hypothesis: 本地修复可能有效，但上游采纳需要维护者和发布证据
  primary_claim_to_prove: 该开源贡献是否被上游接受并进入可用版本
  category_route:
    primary_library: engineering_software_product
    high_risk_regulatory_tags:
      - intellectual_property
  selected_lenses:
    - evidence_contract
    - execution_validation
    - operation_handoff
    - transfer_boundary
  domain_specific_questions:
    - question: issue 是否可复现，PR 是否通过 CI、review、merge 和 release？
      why_it_matters: 防止把本地能跑或 PR 打开外推为上游采纳
      evidence_needed: issue 链接、复现脚本、CI 记录、review 意见、merge commit、release note
    - question: 修复涉及的许可证、依赖和维护责任是否清楚？
      why_it_matters: 开源贡献不只看代码，还要看许可和维护边界
      evidence_needed: LICENSE、依赖清单、maintainer comment、后续维护说明
  micro_task_to_execute: 列出从本地修复到上游采纳的 6 个状态
  actual_execution_evidence:
    - local_passed
    - pr_opened
    - ci_passed
    - maintainer_reviewed
    - merged
    - released
  observed_result: 只有进入 merged/released 才能写上游采纳；此前只能写贡献路径进行中
  downgrade_rule: 缺少 merge/release 时，降级为“已提交贡献候选并完成本地验证”
  promotion_decision: use_once
```
