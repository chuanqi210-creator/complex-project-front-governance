# 真实项目新领域构造器三类烟测回归

current_as_of：2026-06-22

## 1. 小题背景

第 168 节把 next_route 从工具/恢复链优化拉回真实项目/新领域构造。本轮必须检验这个 route-back 是否真的执行，而不是只把机器看版的 next_route 改成一个好看的标签。

本小题选取三个当前现实项目来源，覆盖软件/开源、教育/公共服务、研究/数据/敏感人群三类。目标不是新增三个领域模板，而是验证 `new_domain_governance_builder` 能否从真实来源快速生成临时 trace。

## 3. 三类临时 trace

```yaml
new_domain_governance_builder_smoke_test:
  - project: Google Summer of Code 2026
    primary_claim_to_prove: 协议能治理开源软件贡献项目，而不把 GSoC 简化成“写完代码”。
    selected_lenses:
      - execution_validation
      - operation_handoff
      - stakeholder_context
      - transfer_boundary
    domain_specific_questions:
      - 贡献者交付的是 PR、文档、测试、维护权责，还是一次性 demo？
      - mentor、organization admin、community reviewer 各自如何验收？
      - 项目结束后代码是否进入主分支、发布版本或长期维护路线？
    micro_task_to_execute: 把 GSoC 时间线拆成 proposal -> community bonding -> coding -> midterm evaluation -> final work product -> mentor evaluation 的验收链。
    evidence_needed:
      - 官方时间线和规则
      - 项目 idea / issue / PR / CI / review 记录
      - final work product 和 mentor evaluation
      - 合并、发布或维护交接证据
    downgrade_rule: 只能说完成了一个导师制开源项目交付；除非有合并、发布、用户采用和维护证据，不能外推成开源社区长期影响。
    failure_mode_exposed: mentorship_project_to_software_impact_gap
  - project: 2026 年大学生志愿服务西部计划
    primary_claim_to_prove: 协议能治理公共服务项目，而不把招募规模、协议或服务时长直接当成公共价值。
    selected_lenses:
      - stakeholder_context
      - risk_ethics_permission
      - operation_handoff
      - deliverable_storyline
    domain_specific_questions:
      - 服务对象、服务地、派出学校、接收单位和志愿者各自承担什么风险？
      - 招募协议、服务协议、体检和培训之外，现场支持、转介和退出机制在哪里？
      - 如何证明服务解决了真实需求，而不是只证明有人到岗？
    micro_task_to_execute: 把“报名招募 -> 选拔体检 -> 协议派遣 -> 岗位服务 -> 支持督导 -> 服务评价/续签/退出”拆成公共服务证据链。
    evidence_needed:
      - 官方招募和协议材料
      - 服务岗位需求说明
      - 培训、督导、风险处置和权益保障记录
      - 服务对象反馈与接收单位评价
    downgrade_rule: 只能说完成招募和派遣流程；没有现场需求、服务质量和承接证据时，不能外推成乡村教育或基层治理成效。
    failure_mode_exposed: service_recruitment_to_public_value_gap
  - project: DrivenData Children's Speech Recognition Challenge
    primary_claim_to_prove: 协议能治理研究/数据挑战，尤其能处理儿童语音和 AI 工具数据保留边界。
    selected_lenses:
      - evidence_contract
      - risk_ethics_permission
      - execution_validation
      - transfer_boundary
    domain_specific_questions:
      - 数据是否包含未成年人语音、转写或派生特征，允许在哪些环境中处理？
      - 外部模型、外部数据和 AI 工具的数据传输/保留边界是什么？
      - 竞赛指标能否支持真实儿童语音教育或辅助技术效果？
    micro_task_to_execute: 从规则中提取 external data / pretrained model / test-data independence / data security 四个边界，并把模型效果降级为 benchmark evidence。
    evidence_needed:
      - 官方规则与数据使用条款
      - 本地运行环境和数据保留证明
      - 分群体误差、年龄/口音/语音条件鲁棒性
      - 真实使用前的伦理、隐私和人工复核机制
    downgrade_rule: 只能说在竞赛数据和规则边界内提升了特定语音识别任务；不能外推为儿童语音理解、教育公平或临床/辅助技术可用。
    failure_mode_exposed: benchmark_to_sensitive_population_impact_gap
```

## 4. 共性发现

```yaml
common_failure_mode:
  failure_id: route_back_label_without_real_project_trace
  symptom: 机器看版写了“回到真实项目/新领域构造”，但下一轮没有真实来源、临时 trace、小题执行证据和降级规则
  root_cause: meta_tooling_loop_drift 只规定要 route_back，没有规定 route_back 后第一轮必须实际跑 source-backed builder smoke test
  trigger_signal:
    - next_route 包含 real_project_new_domain_builder_regression
    - 但本轮没有 source_url、category_route、selected_lenses、micro_task_to_execute 和 downgrade_rule
  corrected_behavior:
    - 至少做 3 类真实项目烟测，或在用户给定单一项目时做 1 个深度 trace
    - 每个样例必须有真实来源、主张、镜头、小题、证据、降级边界
  detection_check:
    - source_backed_project_count >= 3
    - category_routes_covered >= 3
    - each_sample_has_micro_task_and_downgrade_rule: true
```
