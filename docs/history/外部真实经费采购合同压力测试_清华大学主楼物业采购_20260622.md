# 外部真实经费采购合同压力测试：清华大学主楼中央区物业管理服务采购项目

current_as_of：2026-06-22

样本：中央政府采购网“清华大学主楼中央区物业管理服务采购项目招标公告”。

## 一、为什么选这个样本

本轮沿第 106 节 `next_route` 继续覆盖“经费预算、采购报销、合同合规”类项目。该类项目在真实实践中非常高频：支教、赛事、社团、科研、企业项目、课程活动都可能涉及预算、采购、报销、合同、发票、验收和归档。但它也特别容易被轻轻带过：一张预算表、一份报价、一个采购公告或一张发票，常被误写成“经费合规、采购闭环、合同履约、审计没问题”。

本样本来自中央政府采购网，公开信息足够明确：项目编号、预算金额、最高限价、采购需求、合同履行期限、投标截止、电子采购系统和采购人/代理机构均可核验。它适合压测协议是否能区分：

1. **公告层事实**：采购项目存在，预算和最高限价公开，采购需求和合同期被列明。
2. **程序层事实**：是否完成需求管理、实施计划、评审、中标、合同公告和质疑处理。
3. **履约层事实**：是否签订合同、实际服务、验收、付款、资产或档案归集。
4. **成效层事实**：是否物有所值、服务质量提升、审计无风险或治理能力改善。

## 三、暴露出的失败模式

```yaml
failure_mode:
  id: F66-01
  name: budget_quote_contract_gap
  trigger: 用户要求写经费预算、采购报销、政府采购、招投标、询价比价、供应商选择、合同签订、项目付款、资产入账、审计归档或财务合规，并希望证明“预算合理、采购合规、合同闭环、可报销、物有所值或审计没问题”
```

## 四、为什么不是 numeric、enterprise 或 governance trace

这个样本有金额、服务采购、合同期和学校治理背景，容易误路由到既有模块：

1. 不是单纯 `numeric_calculation_trace`：预算金额不是数学结论，关键是预算依据、采购方式、合同履约和报销审计。
2. 不是单纯 `enterprise_delivery_trace`：供应商未来可能交付物业服务，但本轮最需要追踪的是采购程序和合同合规，而不是企业产品交付能力。
3. 不是单纯 `student_community_governance_trace` 或 `service_impact_trace`：物业服务可能影响校园运行，但采购合规不能自动证明服务质量或治理成效。
4. 不是普通“项目管理”：政府采购有法定节点，预算、需求、采购文件、评审、合同、验收、付款和信息公开必须分层。

## 六、清华物业采购样本中的最小结论门

1. **只证明项目真实存在**：需要中央政府采购网公告链接、项目编号和公告日期。
2. **证明预算和限价公开**：需要公告中的预算金额和最高限价；不能据此证明价格合理。
3. **证明采购需求公开**：需要公告和完整招标文件；公告摘要不足以证明需求管理和实施计划已审查通过。
4. **证明采购程序推进**：需要投标、开标、资格审查、评审过程、中标成交公告和质疑投诉记录。
5. **证明合同有效和履约开始**：需要正式合同公告、合同条款、服务期、交接记录和履约责任矩阵。
6. **证明履约合格和可付款报销**：需要验收书、服务考核、发票、付款审批、预算支付和资产/档案归集。
7. **证明审计无重大风险**：需要审计或内部复核证据，而不是“没人提出异议”的沉默证据。
