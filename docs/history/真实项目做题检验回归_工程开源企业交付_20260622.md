## 二、三道小题

### R85-01 工程实践：北京市工创赛

```yaml
micro_task_execution_check:
  task: 为“智能物流搬运/智能救援”等赛项填写最小 prototype_validation_trace
  input_material: S1 北京市教委通知
  action_taken: 抽取竞赛命题、赛项、赛程和可支撑主张，列出不能外推的工程结论
  actual_execution_evidence:
    - 页面含“智能物流搬运、智能救援、企业运营仿真、智能产线设计与仿真、未来技术探索和工程文化共7个赛项”
    - 页面含“竞赛分预赛、初赛、决赛三个环节”
  observed_result: 官方通知能证明竞赛组织和赛项存在，不能证明参赛样机稳定、可靠、可部署或可量产
  pass_fail: pass
  remaining_gap: 若要写工程可靠性，仍需队伍测试记录、重复实验、故障样例、仪器/场地条件和安全控制
```

降级表述：

| 想写的结论 | 允许表述 |
| --- | --- |
| 样机具备工程可靠性 | 在竞赛条件下围绕指定赛项形成工程实践作品，可靠性仍待重复测试 |
| 可真实部署到物流/救援场景 | 已完成面向物流/救援主题的竞赛设计或原型任务，真实部署边界待验证 |
| 获奖即可证明工程能力 | 获奖只能说明在竞赛规则下表现较好，不能替代工程验收 |

暴露失败模式：`prototype_demo_gap`。

## 四、协议回写要求

1. 主协议必须保留 `real_project_pressure_test_gate`、`micro_task_execution_check`、`superpower_subagent_orchestration`、`anti_protocol_bloat_gate` 四个命名闸门。
2. 路由表必须把这四个闸门作为条件触发检查，而不是新增一堆用户可见模板。
3. 低摩擦入口必须允许用户用自然语言触发，例如“真实项目做题检验”“开子代理”“先跑一个小题”“别膨胀”。
4. 机器看版必须记录子代理策略；本轮实际使用了两个只读子代理，一个做来源与小题核验，一个做协议落点审查。
