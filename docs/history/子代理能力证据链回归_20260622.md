# 子代理能力证据链回归 2026-06-22

## 1. 小题背景

协议已经有 `subagent_orchestration_ladder` 和 `subagent_task_contract`，但第 156 节机器看版只写了 `subagents.capability: available`。这句话如果没有证据链，容易把三件不同的事混在一起：

1. 环境上下文里出现了 subagent 名称。
2. `tool_search` 发现了可调用的多代理工具。
3. `spawn_agent` 真实返回了 agent id，且主代理完成了等待、接收、取舍和集成。

本轮做一个小题：真实探测子代理能力，并把能力状态拆成可恢复、可审计的证据链。

## 2. 小题输入与执行证据

```yaml
micro_task_execution_check:
  task: 探测当前 Codex 环境的子代理能力，并区分 listed / tool_found / spawn_attempted / returned / integrated
  actual_input:
    latest_machine_board: "## 156. 当前机器看版"
    existing_machine_state: "subagents.capability: available"
    user_prior_requirement: 希望合适时调用 superpower 子代理能力；无法自行打开时编辑好文本由用户代理执行
  actual_execution_evidence:
    - tool_search_query: "multi-agent subagent spawn dispatch parallel agents tools"
    - tool_search_result: exposed multi_agent_v1.spawn_agent / wait_agent / send_input / close_agent
    - spawn_agent_called: true
    - spawned_agent_id: "019eedbd-cb41-7443-b41a-50d864e9066f"
    - spawned_agent_nickname: "Copernicus"
    - delegated_role: read_only_audit
    - delegated_scope: 审查主协议、长日志、路由表、发布包和变更清单中的 subagent 能力记录是否过度承诺
    - wait_agent_timeout_ms: 60000
    - wait_agent_status: timed_out
  observed_result: 工具可发现且 spawn 可执行，但子代理结果未在 60 秒内返回；本轮不能把审计结果写成 accepted，只能写 audit_timeout_non_blocking，并由主代理继续按本地证据修复。
  pass_fail: pass

```

## 3. 归纳出的共性规则

```yaml
failure_mode_id: subagent_capability_overclaim
rule:
  trigger_signal:
    - 机器看版写 subagents.capability: available
    - route_event 选择 spawn_subagent
    - 用户要求使用 superpower / subagent
    - 复杂项目具备并行核验或独立审计机会
  required_behavior:
    - 区分 environment_listed、tool_discovered、spawn_attempted、agent_returned、result_integrated
    - available 不能只凭环境里有 subagent 名称；至少要有 tool_discovery 或 spawn_agent callable 证据
    - 调用子代理后必须记录 agent_id、role、task_boundary、wait_status 和 integration_decision
    - 子代理超时但非关键路径时写 audit_timeout_non_blocking；不能写 accepted
    - 主代理必须继续本地验证，不能把最终判断外包给子代理
  evidence_check:
    - subagent_capability_probe.tool_discovered is true when capability is callable
    - if spawn_attempted is true, agent_id and wait_status are present
    - integration_status is accepted / rejected / conflict_recorded / audit_timeout_non_blocking / not_needed
  route_back:
    - 回到机器看版 codex_tool_state.subagents 和 superpower_subagent_orchestration，补证据链
  friction_effect: 用户仍不需要知道字段名；只有登录、授权、付款、敏感上传或外部系统操作才写 manual_action_required
```

## 4. 本轮写回动作

1. 主协议新增 `subagent_capability_probe`：把子代理能力从一句 available 改成证据链。
2. 机器看版模板的 `codex_tool_state.subagents` 增加 `capability_probe`、`agent_ids`、`wait_status` 和 `integration_decision`。
3. 路由表在 `subagent_orchestration_ladder` 之前加入能力探测要求。
4. 发布包能力清单新增 `subagent_capability_probe`，恢复入口推进到第 158 节。
5. 长日志追加第 157 节回归记录和第 158 节当前机器看版。

## 5. 结果边界

```yaml
pass_fail: pass
what_is_proved:
  - 当前环境可通过 tool_search 暴露 multi_agent_v1 子代理工具
  - 当前轮 spawn_agent 返回了真实 agent id 和 nickname
  - 子代理未在 60 秒内返回，因此只能记录 audit_timeout_non_blocking
what_is_not_proved:
  - 不能声称子代理审计意见已被接受
  - 不能声称所有未来环境都能调用 spawn_agent
  - 不能把用户手动代理当作默认路径
```
