# 机械 EOF 追加工具回归 2026-06-22

## 1. 小题背景

连续优化长日志多次出现“新增章节先误插入历史中部，再由 `latest_board_tail_check` 发现并机械搬移到 EOF”的问题。事后扫描有效，但它仍让每轮恢复链多一次修错成本。本轮把这个失败模式前移：新增 `tools/append_eof_section.py`，将“读旧尾部 -> 追加到真实 EOF -> 状态机扫描标题 -> 检查新标题唯一性”做成可重复工具。

## 3. 归纳出的共性规则

```yaml
failure_mode_id: append_eof_tool_missing
rule:
  trigger_signal:
    - 长日志新增编号章节
    - 当前任务要追加新的当前机器看版
    - 过去两轮内出现过 middle_insertion_recovery
  required_behavior:
    - 追加前运行工具扫描旧尾部标题和伪标题
    - 使用 append_eof_section_tool 或等价机械 EOF 追加流程
    - 追加时声明 expect_old_tail、expect_new_tail、expect_new_heading
    - 追加后立即输出 new_tail、新标题计数、pseudo_heading_count 和 latest_pending_count
  forbidden_behavior:
    - 用重复出现的宽上下文锚点追加长日志编号章节
    - 只在事后搬移，不把机械追加作为默认路径
  route_back:
    - 如果 append 工具检测到旧尾部不匹配，先恢复 current_scope_recovery_scan，不继续追加
    - 如果新标题重复，停止并人工审查历史记录作用域
  friction_effect: 不增加用户问题，只把 AI 自己的写入动作工具化
```

2. 用该工具追加第 161/162 节。
3. 主协议将 `append_only_eof_guard` 升级为优先使用 `append_eof_section_tool`。
