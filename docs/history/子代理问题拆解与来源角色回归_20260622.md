# 子代理问题拆解与来源角色回归 2026-06-22

## 一、回归目的

本轮回应“学习网上的方法、skill、插件，提升构建子代理解决问题的能力，并融入 MD”的要求。目标不是在协议里堆更多工具名，而是抽象出通用治理动作：

1. 先拆问题，再决定是否开子代理。
2. 子代理只拿边界清楚的上下文包，不继承整段会话负担。
3. 子代理输出必须经过主代理集成、审查、拒收或冲突记录。
4. 外部来源必须先标明证据角色，不能把项目页面、通知或报道外推成效果证据。

## 二、学习来源

### 2.1 本地 Superpowers skill

已读取并抽取以下本地 skill 的可迁移方法：

1. `superpowers:dispatching-parallel-agents`：一类独立问题一个 agent；任务必须 focused、self-contained、specific about output。
2. `superpowers:subagent-driven-development`：fresh subagent per task；先 spec compliance review，再 code quality review；不要让子代理读完整计划文件，由主代理提供完整任务文本和上下文。
3. `superpowers:requesting-code-review`：review early, review often；reviewer 获取精心构造的上下文，而不是整段历史。
4. `superpowers:verification-before-completion`：不能相信 agent 自报成功，必须独立验证。

### 2.2 外部方法源

1. OpenAI Agents SDK: `https://openai.github.io/openai-agents-python/multi_agent/`
   - 可迁移点：多 agent 编排可以由 LLM 处理，也可以由代码/规则处理；规则化编排更可预测，适合分类后选择固定 agent 或固定审查链。
2. Microsoft AutoGen docs: `https://microsoft.github.io/autogen/stable/user-guide/agentchat-user-guide/tutorial/agents.html`
   - 可迁移点：通用 assistant agent 适合原型和学习，但真实项目要理解 agent 设计选择，避免把“能调用工具”当成工程上可靠的分工。
3. Codex 当前环境 tool discovery：
   - `tool_search` 已发现 `multi_agent_v1.spawn_agent`、`wait_agent`、`close_agent`、`send_input`、`resume_agent`。
   - 本轮实际 spawn 了一个只读审计子代理，用于审查主协议的子代理段落。

### 4.1 子代理审计集成

Dalton 子代理返回状态：`done`。未修改任何文件。

采纳项：

1. 增加 `spawn_preflight`，把是否可 spawn 变成硬闸门。
2. 增加 `dependency_boundary_check`，避免多个问题同根因时盲目并行。
3. 增加 `controller_prepared_context_packet`，要求主代理打包任务文本、行号片段、约束、验收检查和排除项。
4. 增加 `agent_return_status` 和处理规则，区分 `done`、`done_with_concerns`、`needs_context`、`blocked`、`failed`、`timed_out`。
5. 增加 `review_pipeline`，要求 spec/requirements review 先于 quality/risk review。
6. 增加 `subagent_lifecycle_ledger`，每个实际 spawn 必须有一行账本。
7. 增加 `plan_subagent_binding`，明确子代理从 Plan item 派生，只有主代理能更新 Plan/Goal。
8. 增加 `skill_plugin_discovery_record`，默认由主代理做 skill/plugin 发现和指令读取。

未采纳或降级项：

1. “代码 review 的 base/head sha”未写成所有任务强制项，而是放入 review_scope 的思路中：代码任务用 base/head，文档/协议任务用 changed_sections 和 source/candidate version。
2. “worker 并行写入”未鼓励扩大使用，反而写为默认串行；只有 isolated worktree、无共享状态、merge owner 和验证命令齐备时才允许并行。

## 五、写回主协议的通用机制

新增字段/规则：

1. `subagent_problem_decomposition_builder`
2. `source_role_map`
3. `spawn_preflight`
4. `dependency_boundary_check`
5. `controller_prepared_context_packet`
6. `agent_return_status`
7. `review_pipeline`
8. `subagent_lifecycle_ledger`
9. `plan_subagent_binding`
10. `skill_plugin_discovery_record`
11. 失败模式 `subagent_context_dumping_gap`
12. 失败模式 `source_role_overclaim_gap`

晋升理由：

1. 三类项目都暴露了同一问题：来源先要定角色，子代理先要定边界。
2. 该机制不依赖具体领域；工程挑战、社会实践和健康数据挑战都能复用。
3. 它降低用户摩擦：用户不需要知道该派几个子代理，协议自动判断。
4. 它降低幻觉：子代理和来源都不能直接越权生成最终判断。

## 六、边界

本轮没有完整做三个项目的深度方案；它是 5-30 分钟小题回归，用来检验协议面对新领域时的拆题能力。若后续真的进入某个项目，应重新读取最新官方规则、材料和用户目标。
