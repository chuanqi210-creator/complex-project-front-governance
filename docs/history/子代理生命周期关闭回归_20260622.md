# 子代理生命周期关闭回归 2026-06-22

## 1. 小题背景

第 158 节已经把子代理能力从一句 `available` 改成证据链，并记录 Copernicus 子代理返回、主代理部分采纳。但仍有一个生命周期缺口：子代理完成后是否关闭、关闭证据是什么、关闭失败或重复关闭应如何解释，都没有进入机器看版。

这个缺口会影响后续持续任务：完成子代理如果没有关闭或没有记录关闭状态，可能占用并发槽；如果关闭了但没记录，恢复者会误以为还要等待或继续集成。

## 2. 小题输入与执行证据

```yaml
  task: 复查已完成子代理 Copernicus 的生命周期关闭状态
  target_agent:

    nickname: Copernicus
    role: read_only_audit
  prior_evidence:

  current_execution_evidence:

  observed_result: 子代理不再可由 close_agent 操作；结合上一轮 close_agent 返回 previous_status.completed，当前状态应记录为 closed_or_not_resumable_after_prior_close，而不是 open、running 或 pending。
  pass_fail: pass
  remaining_gap: 协议需要标准化 close_agent_status / cleanup_decision / repeated_close_result，避免完成子代理在恢复链里悬空。
```

## 3. 归纳出的共性规则

```yaml
failure_mode_id: subagent_lifecycle_cleanup_gap
rule:
  trigger_signal:
    - spawn_attempted is true
    - agent_returned / timed_out / blocked / no_longer_needed
    - integration_decision 已写入机器看版
  required_behavior:
    - 如果子代理不再需要，调用 close_agent 或记录为什么不能关闭
    - 机器看版必须记录 close_attempted、close_status、closed_agent_ids、previous_status_summary 和 repeated_close_result
    - close_agent 返回 not found 时，不能自动声称成功；必须结合 prior_close_evidence 判断为 already_closed_or_not_resumable 或 unresolved
    - 未关闭且仍可能占用资源时，写 open_agent_cleanup_required，而不是 silent pass
  evidence_check:
    - spawn_attempted true implies lifecycle_cleanup exists
    - lifecycle_cleanup.cleanup_decision in [closed, already_closed_or_not_resumable, keep_open_with_reason, cleanup_unresolved]
  route_back:
    - 回到 codex_tool_state.subagents.lifecycle_cleanup 和 superpower_subagent_orchestration lifecycle 字段补证据
  friction_effect: 不增加用户输入；只有外部系统需要用户关闭或授权时才写 manual_action_required
```

## 4. 本轮写回动作

1. 主协议新增 `subagent_lifecycle_cleanup` 工具型规则。
2. 机器看版模板的 `codex_tool_state.subagents` 增加 `lifecycle_cleanup`。
3. `superpower_subagent_orchestration` 增加 close/cleanup 字段。
4. 路由表加入子代理完成后的关闭记录要求。
5. 发布包恢复入口推进到第 160 节。
6. 长日志追加第 159 节回归记录和第 160 节当前机器看版。
