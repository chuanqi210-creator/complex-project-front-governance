# 持续治理协议发布包 2026-06-22

用途：把宏观重构后的 Complex 项目持续治理协议、低摩擦入口、真实项目经验库、路由表、压力测试产物和恢复入口整理成可复用结构。当前版本已从“场景覆盖式 trace 目录”升级为“新领域构造式治理内核”，并将第 5 节阶段流程升级为 `dynamic_stage_controller` 动态调度；协议定位也从“启动前置”扩展为覆盖持续推进、Loop 小闭环、评分迭代、协作拓扑、恢复链和交付边界的项目治理规则。

Complex Runtime Kit / 持续治理运行时套件是本发布包的可选落地层：`templates/` 提供 state、evidence、decision、search、question、loop 和 delivery 记录模板，`.codex/shared-skills.json` 提供项目级能力候选清单，`docs/runtime-skill-management.md` 说明能力选择、拒绝、试用和写回规则。Runtime Kit 用于帮助新项目快速启动可恢复的运行记录，不替代主协议，也不新增 verifier required 字段。

## 一、从这里开始

用户入口：

`/Users/chuchenqidawang/Documents/complex-project-front-governance/protocol/Complex项目持续治理_低摩擦用户入口_20260622.md`

推荐用户只说：

```text
这是一个重大项目。
目标是：……
已有材料在：……
我希望结果达到：……
请按 Complex 项目持续治理协议推进，但不要把流程负担转嫁给我。
```

AI 应做：

1. 先启用 `dynamic_stage_controller`，判断 light / standard / deep / continuous 深度和 route_event，并用 `routing_decision_log` 记录未选路线、拒绝理由、触发证据和用户摩擦影响。
1.1. 如果任务是继续优化本协议或 AI 协作流程，先启用 `continuous_optimization_meta_project_profile`，把它当成一个真实项目压缩治理，并用真实小题、反膨胀决策和停止条件收束。
2. 自动识别最终要证明的主张。
3. 启用 `new_domain_governance_builder`，生成本轮临时领域 trace。
4. 用 `category_experience_router` 选择一个主类和必要的 high_risk_regulatory 标签。
5. 从经验库索引抽取 1-3 个相近失败模式校准，而不是复制旧 trace 目录。
6. 用 `agentic_orchestration_capability_builder` 判断是否需要学习 skill/plugin/外部方法、是否有可调用子代理能力、上下文包和 review gate 是否准备好。
7. 用 `agent_topology_selection_trace` 判断本轮应采用 single_agent、agents_as_tools、handoff、parallel_evidence_lanes、orchestrator_workers、reviewer_loop、group_chat 还是 user_proxy，并写清控制权、状态边界、上下文工程、成本收益和降级路线。
8. 用 `subagent_orchestration_pattern_router` 判断单代理是否足够、应选择哪种编排模式、拒绝了哪些模式，以及外部方法如何映射为 work unit。
9. 用 `subagent_orchestration_ladder` 判断是否需要只读审计、并行探索、worker、reviewer 或 inline_fallback。
9. 最多问 1-2 个高杠杆问题，其余治理写入文件。
