# 持续治理协议发布包 2026-06-22

用途：把宏观重构后的 Complex 项目持续治理协议、低摩擦入口、真实项目经验库、路由表、压力测试产物和恢复入口整理成可复用结构。当前版本已从“场景覆盖式 trace 目录”升级为“新领域构造式治理内核”，并将第 5 节阶段流程升级为 `dynamic_stage_controller` 动态调度；协议定位也从“启动前置”扩展为覆盖持续推进、Loop 小闭环、评分迭代、协作拓扑、恢复链和交付边界的项目治理规则。

Complex Runtime Kit / 持续治理运行时套件是本发布包的可选落地层：`templates/` 提供 state、evidence、decision、search、question、prompt、framing、argument、judgment、loop 和 delivery 记录模板，`docs/examples/` 提供填好的 evidence_fill、model_discovery 与 independent_review 黄金样例，`.codex/shared-skills.json` 提供项目级能力候选清单，`docs/runtime-skill-management.md` 说明能力选择、拒绝、试用和写回规则，`docs/complex_new_agent_5_minute_quickstart_20260702.md` 提供新代理低摩擦入口，`docs/behavior_transcript_review_rules_20260702.json` 与 `tools/review_behavior_transcript.py` 提供真实回复 transcript 审查入口。Runtime Kit 用于帮助新项目快速启动可恢复的运行记录，不替代主协议，也不新增 verifier required 字段。

## 一、从这里开始

用户入口：

`/Users/chuchenqidawang/Documents/complex-project-front-governance/protocol/Complex项目持续治理_低摩擦用户入口_20260622.md`

推荐用户用贴合项目的短指令开始：

```text
目标是：……
已有材料在：……
我希望结果达到：……
如果研究框架或模型还没定，请先按模型发现型处理，不要早收敛。
请按 Complex 项目持续治理协议推进，但不要把流程负担转嫁给我。
```

AI 应做：

0. 先用 `complex_behavior_kernel` 压缩本轮行为主线：恢复真实状态、判断项目性质、划清决策权、抓一个最高杠杆问题、选择最轻有效动作、按交付对象输出、留下 next_route 和恢复线索。
1. 再启用 `dynamic_stage_controller`、`project_nature_router` 和 `adaptive_judgment_controller`，判断 light / standard / deep / continuous 深度、evidence_fill / model_discovery / mixed / execution_delivery 项目性质、judgment_mode、autonomy_level、decision_right 和 route_event，并用 `routing_decision_log` 或 `route_evaluator_reflection_gate` 记录未选路线、拒绝理由、触发证据、误判风险、回滚路线和用户摩擦影响。
1.1. 如果任务是继续优化本协议或 AI 协作流程，先启用 `continuous_optimization_meta_project_profile`，把它当成一个真实项目压缩治理，并用真实小题、反膨胀决策和停止条件收束。
1.2. 如果用户要求完整扫描 Complex、连续节拍、多线程/子代理、Plan 模式或 Goal 模式，先启用 `protocol_scan_sequence`、`scheduled_topology_capability_review`、`per_round_goal_lifecycle_gate` 和 `goal_refresh_gate`，说明读取顺序、关键词触发、事件触发优先的复查点、每拍窄 Goal 和 stale/blocked tool goal 处理。
1.3. 如果用户要求先扫描 Complex、设计项目 prompt、确认后再执行，先启用 `complex_prompt_bootstrap_gate` 和 `templates/prompt.md`，输出可复制执行提示词和 execution_bridge；确认前不进入业务执行。
1.4. 如果用户说明模型、研究框架、解释路径或指标口径未定，先启用 `anti_premature_convergence_gate`、`ibis_argument_map_gate` 和 `thought_search_gate`，保留候选框架、问题-观点-论据图和可区分探针，再决定是否进入证据填充。
1.5. 如果用户说“强自治+护栏”“让 AI 自行判断细节”“动态推进”“只在高风险时问我”或类似表达，采用 `decision_rights_matrix`：AI 自行处理可逆、低副作用、项目内的计划、Loop、证据深度、工具取舍、临时分工、长期线程职责微调和发散/收敛节奏；主目标、授权、不可逆动作、交付公开口径和高风险主张变化必须回问。
2. 自动识别最终要证明的主张；模型发现型任务先识别要定义的问题，不把第一个资料缺口当成最终主张。
3. 启用 `new_domain_governance_builder`，生成本轮临时领域 trace。
4. 用 `category_experience_router` 选择一个主类和必要的 high_risk_regulatory 标签。
5. 从经验库索引抽取 1-3 个相近失败模式校准，而不是复制旧 trace 目录。
6. 用 `agentic_orchestration_capability_builder` 判断是否需要学习 skill/plugin/外部方法、是否有可调用子代理能力、上下文包和 review gate 是否准备好。
7. 用 `agent_topology_selection_trace` 判断本轮应采用 single_agent、agents_as_tools、handoff、parallel_evidence_lanes、orchestrator_workers、reviewer_loop、group_chat 还是 user_proxy，并写清控制权、状态边界、上下文工程、成本收益和降级路线。
8. 用 `subagent_orchestration_pattern_router` 判断单代理是否足够、应选择哪种编排模式、拒绝了哪些模式，以及外部方法如何映射为 work unit。
9. 用 `subagent_orchestration_ladder` 判断是否需要只读审计、并行探索、worker、reviewer 或 inline_fallback。
9. 最多问 1-2 个高杠杆问题，其余治理写入文件。
