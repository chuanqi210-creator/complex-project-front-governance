# 前置治理协议真实项目经验库索引 2026-06-22

用途：承接从主协议主体移出的旧具体领域 trace。经验库不替代新项目治理，也不要求用户阅读；它只为 `new_domain_governance_builder` 提供可追溯样例、失败模式、可迁移问题和降级表述。

最新协议状态：主协议第 5 节已由“线性阶段清单”升级为 `dynamic_stage_controller`，并补入 `agent_topology_selection_trace`。经验库仍只提供失败模式和可迁移方法；新项目启动时先由动态阶段控制器决定 light / standard / deep / continuous 深度、route_event、代理拓扑和子代理策略，再按需调用本经验库。

## 一、索引规则

每个经验条目至少保留：

```yaml
experience_library_entry:
  trace_id:
  primary_library:
  risk_tags:
  failure_mode_id:
  source_pressure_test_file:
  current_as_of:
  reusable_method:
  non_transferable_detail:
```

六类路由：

1. `research_data_model`：研究、数据、模型、实验、调研、评价、算法治理。
2. `engineering_software_product`：工程、软件、产品、开源、原型、企业交付。
3. `education_practice_public_service`：教育、社会实践、公益、公共服务、志愿服务。
4. `organization_business_governance`：组织、商业、政策、程序、公平、验收、资源配置。
5. `communication_culture_design`：传播、文化、设计、科普、舆情、国际交流。
6. `high_risk_regulatory`：横切标签，不与前五类互斥；用于健康、安全、法律、隐私、未成年人、财务、食品、交通、心理、知识产权等风险。

## 二、共性失败模式

| failure_mode_id | 共性误判 | 通用修正 |
| --- | --- | --- |
| `output_effect_confusion` | 把照片、作品、demo、活动、证书、播放量、新闻当成效果 | 用 `claim_ladder` 区分输出、过程、结果和长期影响 |
| `one_pass_reliability_gap` | 把一次通过、验收、无事故、PR 合并、比赛完成当成长期可靠 | 用 `execution_validation` 追踪重复、失败样例、真实运行和维护 |
| `governance_document_reality_gap` | 把制度、名单、公示、协议、备案当成治理有效 | 用 `operation_handoff` 和 `stakeholder_context` 追踪责任、申诉、整改 |
| `sample_to_population_gap` | 把样本、访谈、问卷、课堂反馈外推成总体事实 | 用 `evidence_contract` 追踪抽样、偏差、测量和因果边界 |
| `goodwill_permission_gap` | 把公益善意或组织安排当成授权与保护 | 用 `risk_ethics_permission` 追踪知情同意、退出、隐私和转介 |
| `simulation_competition_reality_gap` | 把竞赛、模拟、样机、仿真、企业赛题直接转成现实能力 | 用 `transfer_boundary` 写清从赛题到真实部署缺哪层证据 |
| `authority_source_overclaim` | 把政策、通知、新闻、专家意见当成现场证据 | 用 `evidence_contract` 记录来源层级、时效和可支撑范围 |
| `expectation_only_regression_gap` | 协议优化只写期望行为，没有实际做小题 | 用 `micro_task_execution_check` 留下输入、证据、结果和 remaining_gap |

## 四、调用方式

当新项目启动时：

0. 先恢复或填写 `dynamic_stage_controller`、`gate_activation_matrix` 和 `routing_decision_log`，决定当前是 light / standard / deep / continuous、哪些 gate 被风险触发、是否需要子代理。
1. 再用 `new_domain_governance_builder` 明确最终主张。
2. 用 `category_experience_router` 选择 1 个主类和必要的 high_risk_regulatory 标签。
3. 从本索引中抽取 1-3 个相近经验条目，只复用失败模式、证据缺口和降级方法。
4. 用 `micro_task_execution_check` 做一个小题，而不是复制旧模板。
5. 若本轮学习网上方法、skill、plugin 或子代理模式，先填 `method_transfer_matrix`；只有能生成 work unit、验收证据和降级路线的方法才写回主协议。
6. 若本轮发现的新规则只服务单一领域，写入本经验库；只有跨多个项目反复成立，才走 `rule_promotion_mechanism` 写回主协议。
