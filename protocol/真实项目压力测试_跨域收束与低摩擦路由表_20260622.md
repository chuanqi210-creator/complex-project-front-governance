# 真实项目压力测试：新领域构造式低摩擦路由表

current_as_of：2026-06-22

用途：把外部真实项目练习从“具体 trace 场景目录”收束为“新领域治理构造器 + 大类经验库 + 工具型闸门”。用户只描述项目，AI 自动构造本轮必要镜头，不再要求用户从几十个 trace 名称中选择。

## 一、核心路由

### 1.1 常驻检查

任何重大项目都默认做：

1. `current_basis`：当前依据、非依据、缺口。
2. `evidence_matrix`：核心判断有来源。
3. `decision_log`：重大取舍可追踪。
4. `traceability_matrix`：用户诉求不丢失。
5. `new_domain_governance_builder`：面对新领域时先构造临时 trace。
6. `anti_protocol_bloat_gate`：防止经验库继续膨胀成主协议目录。
7. `dynamic_stage_controller`：先判断 light / standard / deep / continuous 深度和 route_event，再决定推进、压缩、加深、并行、回退或询问用户。
8. `gate_activation_matrix`：把“默认做”的常驻检查拆成 always_minimum、risk_based、continuous_only 和 recovery_only；每个 risk_based gate 必须有 activation_predicate、minimum_by_mode 和跳过记录，避免全量硬跑或低摩擦误跳过。
8.1. `capability_discovery_cadence_gate`：在 `dynamic_stage_controller` 之后、执行前显性完成初始能力盘点，并在阶段切换、用户新增要求、阻塞/验证失败、外部写入或子代理边界、最终主张前复盘 skill / tool / plugin / connector / API / 外部方法；小型本地任务可写 intentionally_skipped，但必须说明已有本地工具足够。
8.2. `continuous_optimization_meta_project_profile`：当任务是继续优化本协议、流程或 AI 协作方式时，把它视为一个真实项目，但只按压缩 profile 跑 stage_0 / stage_1 / stage_2 / stage_4 / stage_7 / stage_8 / stage_9；候选优化点必须标注 promote_to_protocol、add_to_experience_library、reject 或 backlog，不直接新增硬字段。每轮还必须填写 `iteration_quality_bar`，用 entry/execution/exit 判断是否能进入下一轮。
8.3. `trigger_based_continuation_policy`：连续优化默认按触发续跑。只有用户明确指定、新真实项目缺口、验证红灯或高价值 backlog 项成立时才开启新轮；无触发时只保留 backlog，不追加新机器看版。
9. `routing_decision_log`：记录动态路由选项、未选路线、拒绝理由、触发证据、用户摩擦影响和重评条件，防止动态决策变成不可复盘的自由裁量。
10. `subagent_problem_decomposition_builder`：需要来源核验、并行探索、执行 worker 或 reviewer 时，先拆出 independent_work_units、input_context_packet、output_contract、acceptance_evidence、review_gate 和 lifecycle_cleanup_plan；不能把整段会话或大目录直接丢给子代理。
11. `subagent_orchestration_ladder`：需要来源核验、并行探索、执行 worker 或 reviewer 时，先写边界和集成规则；调用前先跑 `subagent_capability_probe`，区分 environment_listed、tool_discovered、spawn_attempted、wait_status 和 integration_decision；调用后若子代理不再需要，必须补 `subagent_lifecycle_cleanup`，记录 close_status、previous_status_summary、repeated_close_result 和 cleanup_decision；不需要时记录 no_spawn_reason。
12. `skill_plugin_learning_loop`：用户明确提 skill/plugin、任务触发本地 skill/plugin，或本轮要学习网上方法提升 agent/subagent/tool 能力时，记录发现问题、读取状态、硬约束、小题试用、采纳决定、写回位置、验证器覆盖和恢复链引用。
13. `skill_plugin_project_fit_trace`：学习 skill/plugin 后必须说明它改善了当前真实项目的哪条主张、阶段缺口、验收链、协作方式或交付证据，并写清前后差异、验收证据和不能迁移边界。
14. `best_practice_learning_contract`：学习网上方法、官方资料、行业最佳实践或 agent 编排方法时，必须把 stage_gap、抽取方法、失败模式、应用测试和不可迁移边界写清楚；只列框架名不算通过。
15. `capability_type_and_side_effect_gate`：区分 skill、plugin、connector、local_tool、web_source、external_method 的操作模式和副作用；涉及安装、授权、外部写入或账号权限时必须显性授权或降级。
16. `external_state_write_guard`：任何可能写入外部系统、改变账号/课程/仓库/成绩/名单/发布状态的能力，都必须记录权限范围、目标对象、敏感数据、子代理能力边界、沙盒/干跑证据、回滚方案和用户授权；缺项则只读或 manual_action_required。
17. `skill_plugin_discovery_gate`：触发 skill/plugin/tool 时记录候选集、选择与拒绝理由、instruction_read_evidence、capability_resolution_order 和 call_or_skip_decision；environment_listed != callable，installable != authorized。
18. `integration_lifecycle_gate`：集成/互通/兼容/生产部署主张必须检查同一版本部署的端到端证据、维护/退役/停止新用户/迁移/支持路线图信号；单边标准能力不能相加为组合可用。
19. `transactional_integration_consistency_guard`：支付、退款、物流、云资源、IoT、库存、工单或 webhook/event/callback 等异步外部副作用，必须检查幂等、重复事件、重试/重放、乱序、权威状态源、对账、test/live 边界和高影响动作补偿。
20. `agentic_orchestration_capability_builder`：学习网上方法、skill、插件或提升子代理能力时，先构造能力缺口、来源组合、method_transfer_matrix、Codex 工具探测、上下文包、验收证据、review gate、主代理集成和生命周期清理，再决定是否 spawn。
21. `agent_topology_selection_trace`：学习外部 agent/skill/plugin 方法后，先决定本轮是 single_agent、agents_as_tools、handoff、parallel_evidence_lanes、orchestrator_workers、reviewer_loop 还是 user_proxy，写清最终答案控制权、状态写入边界、上下文包、并行依赖图、成本收益和降级路线。
22. `subagent_orchestration_pattern_router`：学习外部 agent/skill/plugin 方法后，先判断单代理是否足够，再选择 routing、agents_as_tools、parallel_evidence_lanes、orchestrator_workers、evaluator_optimizer、handoff、reviewer_loop、group_chat 或 code_or_rule_router，并记录 rejected_patterns、method_to_work_unit_mapping、role_selection_reason、context_packet_completeness 和 minimum_application_test。
23. `read_only_audit_subagent_contract`：只读审计子代理禁止编辑、安装、认证、外部写入、Plan/Goal 操作和最终裁决；输出固定为读到的锚点、证据缺口、不能支持的主张、建议补法和主代理需补查项。
24. `desired_state_reconciliation_guard`：Terraform/GitOps/Kubernetes controller/operator 等声明式期望状态系统必须区分 desired source、observed state、plan/diff、drift、锁/单写者、destroy/prune/replace、import/adoption、policy gate、rollback/forward fix 和 operator handoff；不能把可 apply/sync 外推为长期生产安全。
25. `stateful_data_migration_guard`：数据库 schema、索引/约束、字段重命名、类型变更、回填、批处理迁移或数据模型重构必须检查兼容窗口、expand/backfill/cutover/contract 顺序、在线 DDL 锁/超时、索引/约束验证、回填幂等与校验、回滚或前滚边界、观测暂停恢复和 owner runbook。
26. `security_incident_response_guard`：安全漏洞、凭证泄露、扫描告警、PoC、issue/PR、CVE/GHSA/advisory、协调披露或真实安全事件主张必须区分线索、可利用性、active exploitation、compromise、data access/exfiltration、修复部署验证和披露/通知状态；缺事件证据链、遏制修复、凭证轮换、影响范围、监控复发或复盘 owner 时必须降级。
27. `data_artifact_lineage_freshness_guard`：数据集、报表、模型、图表、指标、BI 看板或分析输出支撑结论、下游引用、发布复用或决策时，必须记录产物身份、版本、来源快照、输入 hash、生成/刷新时间、转换逻辑、run/job、代码/参数/环境、lineage、freshness、质量验证、checksum、复现配方、下游引用和 stale 降级；每个关键字段必须标注 `field_evidence_status`，区分 direct_observed、derived_from_metadata、inferred、unknown、not_applicable，不能把静态描述、截图、文件时间、DAG 成功、registry 版本或空字段单独证明当前可决策。
28. `subagent_method_learning_trace`：学习网上方法、skill 或插件来提升子代理能力时，把本地 skill、外部官方/可信方法和 Codex 当前工具能力合成独立问题域、上下文隔离、handoff/guardrail、review/integration 规则；不得只堆框架名。
29. `subagent_result_coverage_gate`：子代理返回后逐项核对 contract_items、returned_items、unverified_items、out_of_scope_items 和主代理补查项。
30. `main_agent_integration_review`：主代理必须记录 conflicts_found、adopted_claims、rejected_claims、final_claim_changed 和降级理由。
31. `claim_readiness_ladder`：每个最终主张绑定 idea、source-backed、local、small-loop、pilot 或 production/public claim 层级，禁止越级表达。
32. `parallel_evidence_lane_trace`：根因/修复高度耦合但证据源可独立核验时，只并行 `read_only_audit` 证据泳道；每条 lane 只交付事实、候选假设和缺口，禁止最终裁决、写 patch、外部评论或改状态，主代理统一合并判断。
33. `claim_lane_binding`：每条证据泳道必须绑定 `claim_ladder` 层级，写明能支持和不能支持的主张；活动量、制品产出、地图覆盖、代码提交、课程场次或问卷样本不能直接证明 uptake、outcome 或 long_term_impact。
34. `deployment_readiness_gate`：当结论从 benchmark、demo、论文指标、样机、离线测试或内部验收上升到真实使用、部署、临床/教育/工程/生产决策或商业交付时，必须补外部有效性、工作流/用户、风险安全授权、监管政策边界、监控承接证据。
35. `source_authority_precedence_trace`：多个来源覆盖同一规则、日期、要求、程序或事实主张，或某来源自称 authoritative / official / sole source 时，记录 claim_scope、precedence_order、conflict_rule、freshness_or_version_check 和降级处理。
36. `current_scope_recovery_scan`：恢复链扫描先区分当前入口、EOF 最新看版和历史记录；历史旧 route 只作为 archive_hit，不单独判定 current stale；旧入口如果缺少历史记录、不再作为当前入口或 archived_recovery_entry 标签，先标为 ambiguous_archive_hit 并修复标签。
37. `latest_board_tail_check`：长期日志最新看板在文件尾。
38. `append_eof_section_tool`：新增长期日志编号章节时，先用工具或等价机械流程扫描旧尾部，再追加到真实 EOF，并检查 expect_old_tail、expect_new_tail、新标题唯一性、pseudo_heading_count 和 latest_pending_count。
39. `rendered_artifact_freshness`：若存在 Word/PDF/图片/网页等最终交付物，检查源稿与渲染物同步、视觉 QA 状态和 `qa_state_transition`；历史 `render_blocked` 修复后必须迁移为带页数、输出目录和抽检证据的 `rendered_pass`。
40. `verify_governance_recovery_tool`：修改后用统一工具核验最新机器看版、发布包、变更清单、route、历史旧入口标签、未闭合 pending、QA 状态一致性和近期晋升的核心字段覆盖；连续自我优化默认使用 `--preset continuous-self-optimization` 和自动旧入口推导，避免每轮手写扫描口径漂移。
41. `change_inventory`：连续迭代修改多个文件时记录恢复入口。
42. `software_delivery_state_boundary_guard`：开源/软件交付流程必须区分 issue triage、PR opened、CI passed、review approved、merged、released、packaged、deployed、real usage、rollback 和 monitoring；缺少同版本 release/deployment/usage 证据时，只能写到对应状态，不能把合并或检查通过外推为生产可用、真实用户采用或长期稳定。

## 二、真实项目压力测试闸门

启用条件：用户要求优化协议、做题检验、扩展题库渠道、横向比较真实项目，或 AI 准备新增规则。

最低通过标准：

1. 先完成 `historical_debt_priority_queue`，确认旧恢复链没有未收口的 stale/pending/manual_action_required。
2. 至少选择 3 类真实项目样本，其中 1 个必须是工程、软件、企业交付、开源维护或真实运行流程类敌意样例。
3. 每个样例必须完成一个 `micro_task_execution_check`。
4. 每个样例必须输出真实来源、最终主张、启用镜头、小题执行证据、失败模式、降级表述和不能外推结论。
5. 如果只列链接、只写优秀案例或只给 expected_behavior，`pressure_test_status` 必须写 `insufficient`。
6. 如果上一轮因 `meta_tooling_loop_drift` 从工具层 route_back，第一轮真实项目回归必须执行 `real_project_builder_smoke_test`：至少覆盖软件/工程、教育/公共服务、研究/数据三类样本；每个样本输出 `source_url`、`category_route`、`selected_lenses`、`micro_task_to_execute`、`evidence_needed`、`downgrade_rule` 和 `failure_mode_exposed`。

## 三、反目录化规则

1. 不得因为经验库里有某个旧 trace，就把它原样展开给新项目。
2. 不得把食品、交通、体育、文化遗产、心理健康、社团治理、赛事承办等具体领域继续写入主协议主体。
3. 高风险监管类是横切标签，不是互斥第六类；例如医疗 AI 同时属于研究/数据/模型和 high_risk_regulatory。
4. 新增规则必须经过 `anti_protocol_bloat_gate`，并说明它是合并、收紧、替代还是进入 backlog。
5. 单一领域细节进入经验库；跨多个项目反复出现、能抽象为通用失败模式的内容，才进入主协议。

## 四、最小用户入口

用户可以只说：

```text
这是一个重大项目。
目标是：……
已有材料在：……
我希望结果达到：……
请按前置治理协议推进，但不要把流程负担转嫁给我。
```

AI 默认只问最多 1-2 个高杠杆问题：

1. 这个项目最终最想证明什么？
2. 是否存在健康、安全、法律、隐私、未成年人、财务、食品、交通、心理、知识产权等高风险边界？

如果上下文足够，AI 直接推断并在 handoff 中记录 assumption。

## 五、执行顺序

1. 先启动 `dynamic_stage_controller`，判断 mode、stage_depth_budget 和 route_event，并写 `routing_decision_log`。
1.1. 同步填写 `gate_activation_matrix`，把后续 gate 分成 always_minimum、risk_based、continuous_only 和 recovery_only；若跳过 risk_based gate，必须写 activation_predicate 未触发的证据。
1.2. 同步填写 `capability_discovery_cadence_gate.initial_scan`：覆盖 local_skills、callable_tools、deferred_tools、plugins_connectors、external_apis_methods，记录 candidates_considered、selected_now、rejected_now、backlog 和 skip_reason；若是小型本地任务可写 intentionally_skipped，但必须说明已有本地工具足够，不强制联网或 tool_search。
1.3. 若本轮任务是“继续优化项目前置协议”，同步填写 `continuous_optimization_meta_project_profile.stage_0` 和 `stage_1`：恢复最新机器看版，说明为什么继续，并把 current_basis / not_current_basis 分成真实跨项目缺口、用户偏好和工具层惯性。
1.4. 若本轮任务是“连续节拍优化”，先填写 `continuous_optimization_meta_project_profile.iteration_quality_bar.entry_gate`：确认最新机器看版、恢复链 failure_count、候选数量、候选最低评分、能力发现状态和 proceed_decision；entry 不通过时只能 stop、ask_user 或 backlog，不进入执行。
1.5. 若最新 `iteration_quality_bar.exit_gate.may_start_next_round` 为 false，且本轮没有用户明确指定、新真实项目缺口、验证红灯或高价值 backlog 项，不得进入新一轮 execution；只能记录 stop / backlog，不追加新编号机器看版。
2. 判断 `primary_claim_to_prove`。
3. 选择一个主类和必要的 high_risk_regulatory 标签。
4. 用 `new_domain_governance_builder` 生成本轮临时 trace。
5. 对每条外部来源填 `source_role_map`，区分项目存在、规则要求、任务定义、权威背景、榜单数据、实施证据和效果证据。
6. 从经验库索引选 1-3 个相近失败模式作校准。
7. 做一个 `micro_task_execution_check`。
7.1. 在 stage_transition、new_user_requirement、blocked_or_failing_verification、external_write_or_subagent_boundary 或 before_final_claim 出现时，更新 `capability_discovery_cadence_gate.periodic_reconsideration`，写 last_checked_at_stage 和 next_check；若候选进入实际调用，再转入 `skill_plugin_discovery_gate` 与副作用门。
7.2. 对连续优化轮次，同步填写 `iteration_quality_bar.execution_gate`：只允许一个主链；小题必须有 source_url、边界化 primary_claim、micro_task 或 no-op 证据、concrete_artifact_or_record_ref、至少一个 direct_observed 状态或字段、至少一个 unknown_or_not_verified 状态或字段、observed_gap、downgrade_rule、claim_readiness_boundary、promotion_recommendation、user_friction_effect 和 anti_protocol_bloat_decisions；缺项时本轮不得 closed。
8. 若本轮触发外部方法、行业最佳实践、skill、plugin 或子代理能力学习，先补 `best_practice_learning_contract`：说明 stage_gap、source_role_map、抽取方法、失败模式、应用测试和不可迁移边界。
9. 若触发 skill/plugin/tool/connector，补 `skill_plugin_discovery_gate` 与 `capability_type_and_side_effect_gate`：区分候选集、选择/拒绝理由、操作模式、副作用、授权边界和是否需要人工操作。
10. 若能力可能写入外部系统、改变仓库/课程/名单/成绩/发布/账号状态，补 `external_state_write_guard`；缺授权、沙盒/干跑、目标对象、敏感数据分类或回滚方案时，降级为只读分析或 manual_action_required。
11. 补 `skill_plugin_learning_loop`：读完必要指令，抽取硬约束，用小题或实际调用验证，再决定 adopt / partially_adopt / reject / backlog。
12. 补 `skill_plugin_project_fit_trace`：把学到的方法绑定到当前项目主张、阶段缺口或验收链，写明前后差异、验收证据、不能迁移边界和复用决定。
13. 若本轮包含集成、互通、标准兼容或生产部署主张，补 `integration_lifecycle_gate`：单边能力不能相加为组合可用；检查同版本端到端证据和维护/退役/迁移/支持信号。
14. 若本轮包含支付、退款、物流履约、云资源、IoT、库存、工单或 webhook/event/callback 异步外部副作用，补 `transactional_integration_consistency_guard`：检查幂等、去重、重试/重放、乱序、权威状态源、对账、test/live 边界和高影响动作补偿。
15. 若本轮目标是提升子代理能力，先补 `agentic_orchestration_capability_builder`：把本地 skill/plugin、外部方法来源和当前 Codex 工具能力转成 `method_transfer_matrix`、能力缺口、上下文包、验收证据、review gate、主代理集成和生命周期清理，再决定是否 spawn。
16. 若本轮目标是提升子代理能力，补 `agent_topology_selection_trace`：在 single_agent、agents_as_tools、handoff、parallel_evidence_lanes、orchestrator_workers、reviewer_loop、group_chat 和 user_proxy 之间选择，写清控制权、写状态边界、上下文工程、并行安全、成本收益、降级路线和小题验证。
17. 若本轮目标是提升子代理能力，补 `subagent_orchestration_pattern_router`：先判断单代理是否足够，再选择编排模式，记录 rejected_patterns、method_to_work_unit_mapping、role_selection_reason、context_packet_completeness、minimum_application_test 和降级规则。
18. 若本轮包含 Terraform/IaC/GitOps/Kubernetes controller/operator、配置管理、环境同步或任何声明式配置驱动真实资源的流程，补 `desired_state_reconciliation_guard`：检查 desired source、observed state、target scope、execution model、plan/diff、drift、锁/单写者、破坏性动作、adoption/import、policy gate、rollback/forward fix 和 operator handoff；缺项时降级为 source-backed、plan-only 或 sandbox validated。
19. 若本轮包含数据库 schema、索引/约束、字段重命名、类型变更、数据回填、批处理迁移或会影响线上读写路径的数据结构变化，补 `stateful_data_migration_guard`：检查兼容窗口、expand/backfill/cutover/contract 顺序、旧新代码混跑、在线 DDL 锁/表重写/超时、索引/约束验证、回填幂等与校验、回滚或前滚边界、feature flag/发布顺序、观测暂停恢复和 owner runbook。
20. 若本轮包含安全漏洞、疑似入侵、凭证泄露、扫描告警、PoC、CTF/writeup、issue/PR、CVE/GHSA/advisory、协调披露或安全修复主张，补 `security_incident_response_guard`：区分 source_type、claim_type、evidence_level、asset scope、授权联系人、分级依据、利用/入侵状态、遏制缓解、凭证轮换、影响范围、修复验证、披露/通知、监控复发和复盘 owner；缺项时把主张降级为线索、候选、未验证修复或披露未完成。
21. 若本轮的数据集、报表、图表、模型、指标、表格或分析输出会支撑结论、下游引用、发布复用或决策，补 `data_artifact_lineage_freshness_guard`：区分 artifact 身份、来源快照、输入 hash、生成/刷新、转换逻辑、run/job、代码/参数/环境、lineage、freshness、质量验证、checksum、复现配方、下游引用和 stale 降级；同时给每个关键字段写 `field_evidence_status`、evidence_uri_or_hash、extracted_value 和 gap_action，避免 guard 字段被当作勾选表。
21.1. 若本轮包含开源 PR、CI、release、package、deployment、production readiness 或真实用户可用性主张，补 `software_delivery_state_boundary_guard`：分别记录 issue_state、pr_state、ci_state、review_state、merge_state、release_state、package_state、deployment_state、usage_state、rollback_or_monitoring_state、claim_readiness 和 cannot_conclude；缺 release/deployment/usage/monitoring 证据时，结论不得越过已验证状态。
22. 若本轮目标是提升子代理能力，补 `subagent_method_learning_trace`：至少合成本地 skill/plugin、外部方法来源和当前 Codex 工具能力，输出独立问题域、上下文隔离、handoff/guardrail、review/integration 规则，并用 `method_transfer_matrix` 记录 fit_signal、reject_signal、generated_work_unit、minimum_application_test 和 non_transfer_boundary。
23. 先运行 `subagent_problem_decomposition_builder` 和 `subagent_dispatch_readiness`，再根据 `subagent_orchestration_ladder` 决定 not_needed、read_only_audit、parallel_exploration、worker、reviewer 或 inline_fallback；read_only_audit 必须使用 `read_only_audit_subagent_contract`；若写 `capability: available`，必须先用 `subagent_capability_probe` 记录 tool_discovered 或 spawn_attempted 证据；若已 spawn 且不再需要，补 `subagent_lifecycle_cleanup`；并把选择写入 `routing_decision_log`。
24. 若有子代理返回，补 `subagent_result_coverage_gate` 和 `main_agent_integration_review`，逐项核对 contract 覆盖、未验证项、冲突、主代理补查和 final_claim_changed。
25. 若同一问题的最终根因/修复高度耦合，但日志、triage、来源、代码阅读或评审标准可独立核验，补 `parallel_evidence_lane_trace`；只允许只读证据泳道，禁止并行 root-cause/fix worker。
26. 每条 evidence lane 必须补 `claim_lane_binding`：说明对应 `claim_ladder` 层级、能支持什么、不能支持什么、需要哪类更强证据才能上升到使用/结果/影响主张。
27. 每个最终主张必须补 `claim_readiness_ladder`；若证据只到 source-backed、locally verified 或 small-loop validated，降级表述不能写成 pilot / production / public impact ready。
28. 若结论上升到真实使用、部署或高风险决策，补 `deployment_readiness_gate`；缺外部有效性、工作流用户、风险授权、监管政策或监控承接证据时，降级为测试条件下表现或进一步验证潜力。
29. 若外部来源之间可能冲突或覆盖同一主张，补 `source_authority_precedence_trace`；冲突未解时先降级，不把摘要页或时间线视图写成最终权威。
30. 写出降级表述和不能外推边界。
31. 若要新增规则，先过 `rule_promotion_mechanism` 和 `anti_protocol_bloat_gate`。
31.1. 对持续优化元项目的每个候选优化点，必须先写 `promote_to_protocol | add_to_experience_library | reject | backlog`。单一真实小题只允许进入经验库或 backlog；至少两个真实项目暴露同类失败模式才考虑主协议规则；能被 verifier 稳定检查时才考虑 required 字段。
32. 若触及最终交付物或历史 QA 缺口，先运行 `rendered_artifact_freshness`，确认 `render_blocked`、`structural_scan_pass`、`rendered_pass` 是否与当前证据一致；阻塞修复后必须写 `qa_state_transition: blocked_to_pass` 和渲染证据。
33. 修改后用 `current_scope_recovery_scan` 确定当前作用域，再运行 `verify_governance_recovery_tool --preset continuous-self-optimization` 或等价综合扫描；全文 rg 命中旧 route 只能作为候选，不能直接当作当前 stale；旧恢复入口缺少历史标签时先修复为 archive_hit 或 current_scope。若本轮刚晋升了核心字段，必须确认 preset required/forbid 已同步，否则记录 `verifier_required_field_drift` 并先修验证器。验证器的 required 字段必须由共享核心源和 scope extras 生成，测试需覆盖生成方式、extras 精确匹配和缺失/多余 scope 失败，避免五份手写 required 列表重新漂移。
33.1. 对连续优化轮次，验证后必须填写 `iteration_quality_bar.exit_gate`：记录同步文件、验证命令、next_round_decision 和 may_start_next_round。只有 exit gate 全部通过且 may_start_next_round 为 true，才进入下一轮。
34. 如果要追加长期日志新编号章节，先运行 `append_eof_section_tool` 或等价机械 EOF 追加流程；禁止用宽上下文锚点作为默认写入方式。追加后再运行 `latest_board_tail_check` 和 `verify_governance_recovery_tool --preset continuous-self-optimization`。
35. 若恢复链验证 `failure_count: 0` 且最近 6 个以上连续机器看版都属于工具、恢复链、子代理生命周期或验证器优化，触发 `meta_tooling_loop_drift` 检查；下一轮默认选择真实项目压力测试、新领域治理构造器或最终交付可用性检查，而不是继续优化工具层。
36. 若 next_route 已指向 `real_project_new_domain_builder_regression` 或持续优化元项目烟测，必须先跑 source-backed smoke test，不能只把 next_route 改成真实项目标签；每个样本必须输出 source_url、primary_claim、selected_lenses、micro_task、observed_gap 和 downgrade_rule。

## 六、下一轮建议

下一轮默认以 `continue_self_optimization_with_iteration_entry_gate_or_stop` 为入口，从 `## 217. 当前机器看版` 恢复。只有用户明确指定、新真实项目缺口、验证红灯或高价值 backlog 项触发时，才进入新一轮；无触发时停止连续优化并只保留 backlog。
