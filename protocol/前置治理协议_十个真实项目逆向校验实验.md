## 0. 实验方法

### 0.1 为什么选竞赛/公开赢家项目

这类项目天然有两个优势：

1. **题面和答案相对分离**：竞赛页提供问题、数据、规则和评价方式；赢家仓库或复盘提供真实高质量解法。
2. **适合测试前置治理**：题面通常包含目标、限制、评价指标、资源边界、提交规则和真实世界意义，正好能测试协议是否能识别 current_basis、not_current_basis、最小闭环和执行前风险。

### 0.2 每个项目固定流程

1. **题面隔离**：只读取竞赛/项目 problem statement、data description、rules、metric，不读赢家方案。
2. **协议推演**：按协议输出 material intake、问题层级、发散、主链、Goal、Plan、评分、最小闭环、执行前闸门。
3. **答案查阅**：读取赢家仓库、官方复盘、论文或案例报告。
4. **差距诊断**：比较“我们按协议会怎么做”和“高质量真实答案实际怎么做”。
5. **协议增量**：只把可迁移经验写回协议，不把个案技巧泛化成普遍规则。

### 0.3 证据等级

| 等级 | 含义 | 能否写回协议 |
| --- | --- | --- |
| E0 | 单个项目的个人观察 | 只能写进实验记录 |
| E1 | 单个项目有明确公开答案支持 | 可写成候选规则 |
| E2 | 2-3 个项目重复出现 | 可写进协议正文 |
| E3 | 多类型项目重复出现且有外部方法支持 | 可写成高优先级规则 |

## 2. P01：Pump it Up: Data Mining the Water Table

### 2.1 题面隔离

只读题面时得到的 current_basis：

| id | source | type | role | current_basis | reason |
| --- | --- | --- | --- | --- | --- |
| P01-M1 | DrivenData 竞赛页 | problem | authoritative input | yes | 明确目标：预测水泵状态 |
| P01-M2 | DrivenData rules | rules | authoritative input | yes | 明确外部数据、提交、赢家开源和文档要求 |
| P01-M3 | 数据字段说明 | data schema | authoritative input | yes | 字段包含地理、建造年份、管理方式、支付、水源、水质等 |
| P01-M4 | winners repo | answer | answer-only | no_before_answer | 题面阶段禁止读取 |

题面事实：

1. 任务是三分类：functional、functional needs repair、non functional。
2. 数据来自 Taarifa 和坦桑尼亚水利部。
3. 现实目的不是单纯分类，而是改善维护，让清洁饮用水可获得。
4. 竞赛规则禁止未明确允许的外部数据。
5. 赢家需要开放代码并提供文档。

### 2.2 按协议进行的前置治理推演

#### 问题层级

| 层级 | 判断 |
| --- | --- |
| 世界问题 | 坦桑尼亚水点状态影响维护和饮用水可获得性，资源有限时需要更早判断哪些水点会坏或需要维修 |
| 经验规格 | 竞赛指标是分类准确率/排行榜；现实规格是维修资源调度是否更有效 |
| 操作问题 | 用给定字段建立三分类模型，避免外部数据，设计交叉验证和提交格式 |
| 表达问题 | 需要同时解释模型效果、关键特征、可复现代码和维护场景意义 |

#### 发散方向

1. **强基线路线**：先做字段清洗、类别压缩、缺失值和时间/地理基础特征，再用随机森林/GBDT。优点是低风险、快闭环。
2. **地理机制路线**：围绕经纬度、region、ward、basin、population、construction_year 建立空间老化和维护机制特征。优点是贴近现实解释；风险是过拟合地区编码。
3. **管理制度路线**：围绕 installer、funder、management、payment、permit、scheme_management 构建治理/维护机制特征。优点是能连接真实水务治理；风险是高基数类别噪声大。
4. **复杂集成路线**：多模型 stacking / blending。优点是可能冲榜；风险是对协议而言过早复杂化。

#### 主链收束

> 在禁止外部数据的边界下，先建立可复现强基线，用时间、地理、高基数类别和管理机制特征提升三分类准确率，再用误差分析判断模型是否真的服务维护优先级。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 数据审计 | train/test/schema | 缺失、异常、类别基数、标签分布 | 统计摘要 | 如果 id 对不上或标签泄漏，暂停 |
| 基线模型 | 原始字段最小清洗 | baseline accuracy | 交叉验证 | 如果低于简单 majority baseline 太多，回退 |
| 特征闭环 | 时间、地理、类别频次、稀有值合并 | feature set v1 | CV 提升 | 如果只提升 public LB 不提升 CV，标记过拟合 |
| 现实解释 | 特征重要性、错误案例 | 维护含义 | 与世界问题对照 | 如果无法解释维护价值，不写成真实影响 |

### 2.3 查阅答案后的真实做法

公开赢家仓库显示，madRid 团队的提交在 2017-01 排名第 5，分数 0.8265；仓库列出了多位领先竞争者提交。其代码中使用的关键做法包括：

1. 读取 training_values、test_values、training_labels，并确认 id 对齐。
2. 将 train 和 test 合并后统一做特征工程。
3. 从 `date_recorded` 构造距离 2014-01-01 的天数和月份。
4. 用经纬度构造地理距离特征。
5. 对 factor 类字段计算按类别出现频次的 `hash` 特征。
6. 将低频类别合并为 `-9999`。
7. 使用 Rborist 随机森林类模型，设定 1000 棵树。

### 2.4 差距诊断

| 我们按协议预期 | 真实答案 | 经验 |
| --- | --- | --- |
| 先做强基线，不急于复杂集成 | 领先方案确实是特征工程 + 强树模型 | 协议应强化 baseline-first，而不是默认“复杂项目需要复杂方案” |
| 关注时间、地理、高基数类别 | 真实答案正好做日期、地理距离、类别频次/稀有值处理 | 题面字段结构已经暗示关键工程，不必等答案才发现 |
| 区分正式指标和现实效用 | 答案主要服务排行榜，较少解释维修调度落地 | 协议应强制记录“正式评分指标”和“真实效用指标”的差异 |
| not_current_basis 记录答案 | 题面规则还明确禁止外部数据 | 协议不仅要有 not_current_basis，还应有 forbidden_basis / resource_boundary |

### 2.5 写回协议的候选规则

当前证据等级：E1。

候选规则：

1. **正式指标 / 真实效用双指标**：所有项目都要区分“评审/排行榜/论文指标”和“现实世界效用指标”。二者不一致时，Plan 必须说明取舍。
2. **资源边界显式化**：Material Intake 除 current_basis / not_current_basis 外，增加 forbidden_basis 或 resource_boundary，记录禁止使用的数据、工具、路线和表达。
3. **baseline-first 闭环**：复杂项目进入执行前，必须设计一个最低复杂度强基线；只有当强基线不足以验证主链时，才升级复杂方案。

## 3. P02：DengAI: Predicting Disease Spread

### 3.1 题面隔离

只读题面时得到的 current_basis：

| id | source | type | role | current_basis | reason |
| --- | --- | --- | --- | --- | --- |
| P02-M1 | DrivenData 竞赛页 | problem | authoritative input | yes | 明确任务：预测每个城市、年份、周的 `total_cases` |
| P02-M2 | DrivenData data / metric / submission 说明 | data schema / metric | authoritative input | yes | 明确两座城市、未来留出、MAE、整数提交格式 |
| P02-M3 | DrivenData rules | rules | authoritative input | yes | 明确数据使用和提交边界 |
| P02-M4 | 官方 benchmark | answer | answer-only | no_before_answer | 题面阶段禁止读取；本轮答案侧使用的是公开官方 benchmark，不是赢家仓库 |

题面事实：

1. 任务是预测 San Juan, Puerto Rico 和 Iquitos, Peru 每周报告的登革热病例数。
2. 预测单位是 `(city, year, weekofyear)`，目标列是 `total_cases`。
3. 测试集是纯未来留出：顺序连续，且与训练数据不重叠。
4. 正式指标是 Mean Absolute Error。
5. 提交文件必须同时包含两座城市的预测，`total_cases` 要以整数表示。
6. 规则要求参与者使用竞赛数据，除非竞赛网站另有允许，不得使用其他数据开发和测试模型。

### 3.2 按协议进行的前置治理推演

#### 问题层级

| 层级 | 判断 |
| --- | --- |
| 世界问题 | 登革热暴发与气候、湿度、温度、季节和城市环境有关；公共卫生系统需要提前感知风险，以便安排防控资源 |
| 经验规格 | 竞赛规格是未来周病例数的 MAE；现实规格是能否更早识别风险窗口、支持干预时机和资源配置 |
| 操作问题 | 用给定气象、遥感、时间和城市字段，做未来时间留出下的病例数预测，避免时间泄漏和外部数据越界 |
| 表达问题 | 需要说明模型不是普通回归练习，而是一个“时间、地点、疾病机制”共同约束的预测问题 |

#### 资源边界

| 字段 | P02 判断 |
| --- | --- |
| allowed_basis | 竞赛提供的训练特征、训练标签、测试特征、数据说明、提交格式、规则 |
| forbidden_basis | 未来标签、测试集答案、未被允许的外部疫情/气候/人口数据、答案 benchmark 在审题阶段的做法、随机切分导致的未来信息泄漏 |
| gray_zone | 如果规则另有允许，外部公共气候或流行病学知识可作为解释背景，但不能未经确认直接进入模型依据 |
| boundary_reason | 竞赛公平、未来预测部署语义、公共卫生问题的可解释性和数据许可 |

#### 发散方向

1. **朴素时间基线路线**：按城市建立历史均值、季节周均值、上一年同周或滚动均值 baseline。优点是能快速识别任务难度；风险是无法利用气象变化。
2. **城市分治路线**：San Juan 和 Iquitos 的地理、气候和病例尺度差异明显，先分别建模，再合并提交。优点是尊重场景异质性；风险是样本更少。
3. **计数模型路线**：`total_cases` 是非负整数，先考虑 Poisson / Negative Binomial 等计数回归。优点是输出形态与目标一致；风险是非线性和滞后效应表达不足。
4. **机器学习路线**：随机森林、GBDT、XGBoost 等模型加入滞后、滚动、季节和湿度温度特征。优点是捕捉非线性；风险是时间切分不严导致虚高。
5. **公共卫生解释路线**：把温度、湿度、降水、植被指数与病例风险联系起来，强调预警和干预含义。优点是连接真实效用；风险是竞赛数据不一定足以证明因果。

#### 主链收束

> 在不使用未授权外部数据、且严格模拟未来预测的边界下，先建立城市分治的泄漏安全 baseline，再根据目标的计数属性选择统计模型或机器学习模型，并同时评估 MAE 与公共卫生预警可用性。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 数据审计 | train features / labels / test features | 城市、年份、周、缺失、标签分布、时间范围 | schema 与时间序列检查 | 如果训练/测试顺序或标签对齐不清，暂停 |
| 切分语义 | 训练集按城市和时间排序 | 严格未来验证集 | 每个城市使用后段时间作为验证 | 如果使用随机切分，不能证明未来预测能力 |
| 基线模型 | 城市历史均值、季节均值、上一年同周 | baseline MAE | 与验证集 MAE 比较 | 如果复杂模型不显著超过基线，不能宣称改进 |
| 计数模型 | 选定气象/湿度/温度变量 | Poisson / Negative Binomial 预测 | MAE 与残差分布 | 如果输出负数或小数处理不一致，回退 |
| 真实效用解释 | 高风险周、误差案例、城市差异 | 预警含义 | 对照世界问题 | 如果只能优化 MAE，不能写成公共卫生解决方案 |

### 3.3 查阅答案后的真实做法

本轮答案来源是 DrivenData 官方 benchmark 教程，而非正式赢家仓库；因此它更适合验证“合理基线和问题理解”，不代表最优竞赛解。

官方 benchmark 的关键做法包括：

1. 明确把数据看成两座城市的问题：San Juan 和 Iquitos 可能有不同传播模式，所以分城市训练模型，最后合并预测。
2. 先分析 `total_cases` 的分布，指出它是非负整数，因此是计数预测问题。
3. 用均值和方差关系判断 Poisson 与 Negative Binomial 的适配性；当方差远大于均值时，采用 Negative Binomial。
4. 做气象变量与病例数的相关性探索，重点观察湿度、露点温度、最低温、平均温等变量。
5. 因为是时间序列模型，训练/验证采用严格未来留出；San Juan 和 Iquitos 分别取前段训练、后段验证。
6. 使用 `statsmodels` 的 Negative Binomial GLM，按 MAE 选择超参数，并将预测转为整数。

答案来源：

- DengAI 题面与数据说明：https://www.drivendata.org/competitions/44/dengai-predicting-disease-spread/page/82/
- DengAI 官方 benchmark：https://drivendata.co/blog/dengue-benchmark

### 3.4 差距诊断

| 我们按协议预期 | 真实答案 | 经验 |
| --- | --- | --- |
| 严格未来切分，避免随机切分 | benchmark 明确使用 strict-future holdout | 协议需要把“验证切分语义”升级为通用闸门 |
| 两座城市可能应分别处理 | benchmark 直接分城市训练再合并 | 协议需要强制检查“一个总模型是否掩盖关键异质性” |
| `total_cases` 是计数目标，应考虑计数模型 | benchmark 以 Poisson / Negative Binomial 作为模型起点 | 协议需要在 Goal/Plan 前识别目标变量形态，而不是只看任务名称 |
| 区分 MAE 和公共卫生效用 | benchmark 主要服务教程和 MAE，未真正落到防控资源配置 | 双指标规则再次被验证：正式指标不自动等于真实效用 |

当前证据等级：

| 规则 | 证据等级 | 理由 |
| --- | --- | --- |
| 正式指标 / 真实效用双指标 | E2 | P01 的排行榜准确率 vs 水务维护效用，P02 的 MAE vs 公共卫生预警效用均重复出现 |
| 资源边界显式化 | E2 | P01 的外部数据禁用，P02 的外部数据、未来标签和时间泄漏边界均重复出现 |
| baseline-first 闭环 | E2 | P01 领先方案从强特征基线出发，P02 官方 benchmark 也从可解释基线模型出发 |
| 验证切分语义 | E1 | P02 强支持，但仍需后续项目继续观察 |
| 场景/群体异质性检查 | E1 | P02 强支持，后续可在城市、学校、用户群、地区、样本来源等项目中继续验证 |

候选规则：

1. **验证切分语义规则**：任何项目的验证方式都必须回答“它模拟的真实部署是什么”。时间预测不能用随机切分证明未来能力；跨地区部署不能只用同地区随机验证证明泛化。
2. **场景/群体异质性规则**：如果数据或任务天然包含城市、学校、地区、人群、设备、平台、时期等分组，前置治理必须判断“一个总方案”是否会掩盖关键差异。
3. **目标形态规则**：在写 Goal 和 Plan 前，必须识别输出目标的形态，例如三分类、计数、连续值、排名、文本、图像分割、结构化行动建议。目标形态决定 baseline 和验证方式。

### 4.1 题面隔离

只读题面时得到的 current_basis：

| id | source | type | role | current_basis | reason |
| --- | --- | --- | --- | --- | --- |
| P03-M1 | DrivenData 竞赛页 | problem | authoritative input | yes | 明确任务：用有限历史数据预测楼宇能源消耗 |
| P03-M2 | DrivenData data / metric / submission 说明 | data schema / metric | authoritative input | yes | 明确历史能耗、天气、节假日、建筑元数据、WRMSE/NWRMSE 和提交格式 |
| P03-M3 | DrivenData rules / data use | rules | authoritative input | yes | 明确禁止外部数据、禁止未来插值、预测时可用信息边界 |
| P03-M4 | winners repo / winners blog | answer | answer-only | no_before_answer | 题面阶段禁止读取；搜索结果如暴露摘要，也不进入题面推演依据 |

题面事实：

1. 任务是从有限历史数据预测楼宇未来能源消耗；现实意义是帮助设施管理、自动化系统、能源效率和运营优化。
2. 数据覆盖 200 多个 building sites，包含历史能耗、天气数据、天气预报、建筑元数据和节假日信息。
3. 预测粒度和周期不统一，题面区分了不同时间步长和时间范围；历史数据粒度与需要预测的粒度一致。
4. 外部数据被禁止，除竞赛论坛明确列出的预训练神经网络例外。
5. 只有能完全复现、且能证明只使用过去数据做预测的算法，才有资格获奖。
6. 明确禁止用未来数据做插值；预测某个 period 时，不能使用该 period 之后的数据。
7. 题面给出 prediction-time data contract：对 building `B_i` 的 forecast period `T_p`，可以使用 `T_p` 之前的本建筑能耗、其他建筑在 `T_p` 之前的数据、提供的天气数据和已经做出的本 period 预测；不能使用本建筑或其他建筑在 `T_p` 期间及之后的真实能耗。
8. 正式指标是按 forecast period 计算的 Weighted RMSE，再用该 period 的平均能耗归一化，最后对所有 building / period 取平均；权重会降低更远未来时间步的重要性。
9. 赞助方还会看某些建筑类别或时间范围上的平均 NWRMSE，鼓励参赛者识别值得单独观察的类别。

### 4.2 按协议进行的前置治理推演

#### 问题层级

| 层级 | 判断 |
| --- | --- |
| 世界问题 | 楼宇能耗预测直接影响设施异常检测、节能策略、设备调度和电力运营；数据少、建筑差异大、未来不可见是主要困难 |
| 经验规格 | 竞赛规格是 NWRMSE/WRMSE；现实规格是能否在不同建筑、不同时间粒度和不同预测周期上稳定服务运营决策 |
| 操作问题 | 在严格 prediction-time availability 边界下，构造只使用过去信息的时间、天气、节假日、建筑和历史聚合特征 |
| 表达问题 | 需要证明算法不是插值，不偷看未来；同时说明哪些建筑/周期可用，哪些场景需要更多数据 |

#### 资源边界

| 字段 | P03 判断 |
| --- | --- |
| allowed_basis | 题面提供的历史能耗、天气/天气预报、节假日、建筑元数据、提交时间戳、规则和 metric |
| forbidden_basis | 未授权外部数据、测试真实能耗、forecast period 期间及之后的其他建筑真实能耗、任何跨 omitted period 的插值、依赖具体年份识别测试洞的特征 |

| boundary_reason | 竞赛公平、真实预测部署、反泄漏、复现资格、运营可解释性 |

#### 验证切分、场景异质性与目标形态

```yaml
validation_semantics: 按 forecast period 模拟未来留出；验证必须重现预测时只能看见过去数据的状态
segment_or_context: SiteId / ForecastId / 时间粒度 / 预测周期长度 / 建筑元数据 / 天气站距离 / 工作日与节假日
target_form: 连续非负时间序列预测
metric_structure: weighted + normalized RMSE；远期步长权重较低，不同建筑用平均能耗归一化
chosen_strategy: 先做泄漏安全的季节/历史基线，再升级到历史聚合特征 + 天气/节假日 + 树模型
what_this_validation_can_prove: 模型是否能在模拟未来留出的 period 上降低 NWRMSE
what_this_validation_cannot_prove: 不能自动证明真实楼宇运营能长期节能，也不能证明跨新建筑无数据时有效
```

#### 发散方向

1. **季节/周期基线路线**：按 SiteId、day-of-week、time-of-day、节假日构造历史均值或上一周/上一日基线。优点是低泄漏、可解释；风险是无法处理天气异常。
2. **prediction-time contract 路线**：为每个目标时间戳定义“此刻可见什么”，所有特征必须通过这个合约过滤。优点是防止插值和未来泄漏；风险是实现复杂。
3. **历史聚合特征路线**：按建筑、ForecastId、时间、星期、节假日和不同历史窗口计算 mean/median/min/max。优点是贴合能耗周期；风险是窗口设计不严会偷看未来。
4. **天气/节假日路线**：连接温度、天气站距离、节假日和工作/休息日。优点是贴近运营机制；风险是天气缺失和站点匹配误差。
5. **模型集成路线**：LightGBM、随机森林、ExtraTrees 或分时间粒度模型。优点是可捕捉非线性；风险是成本高、复现慢、解释弱。
6. **运营分层路线**：按建筑类型、时间粒度、预测周期或能耗水平拆分误差，识别哪些场景需要额外数据。优点是服务真实效用；风险是与排行榜最优不完全一致。

#### 主链收束

> 先建立严格的 prediction-time availability 合约，再用历史周期基线验证可预测性；随后构造只来自过去的历史聚合、天气、节假日和建筑元数据特征，采用与 NWRMSE 对齐的验证与模型训练，同时输出分建筑/分时间粒度误差，判断它是否真的能服务能源运营。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 合约审计 | 题面规则、forecast period、submission timestamps | 每个预测点的可用信息表 | 手工检查 forbidden_basis | 如果某特征依赖未来真实能耗，禁止进入 |
| 基线模型 | SiteId、历史能耗、时间、节假日 | 季节/上一周/历史均值 baseline | forecast period 后段留出 NWRMSE | 如果基线无法复现指标，先不做复杂模型 |
| 指标复现 | validation truth / prediction | NWRMSE 计算器 | 与题面公式对照 | 如果只算普通 RMSE，不能进入执行 |
| 特征闭环 | 历史窗口、天气、节假日、元数据 | feature set v1 | 检查每列是否满足 prediction-time contract | 如果窗口跨越测试洞，回退 |
| 分层诊断 | SiteId / sampling / forecast horizon | 子群体误差表 | 对照赞助方关心的类别 | 如果平均分掩盖关键失败，不能写成真实运营可用 |

### 4.3 查阅答案后的真实做法

答案来源包括 Power Laws winners 仓库、第一名代码和 DrivenData winners 访谈。

公开答案的关键做法包括：

1. winners 仓库 README 明确总结：领先方案普遍是集成模型，并且非常小心地避免时间序列中的未来数据泄漏；常用天气、节假日、合成特征和 LightGBM。
2. 第一名方案说明：先做大量 preprocessing，计算基于历史的聚合函数，为训练集和测试集每个点生成特征，再训练 LightGBM。
3. 第一名 README 还把复现成本写得很具体：建议至少 64GB 内存、8+ 虚拟 CPU、约 10GB 磁盘；在 16 核 CPU 上可能运行到 24 小时。
4. `preprocess.py` 中的聚合特征围绕过去窗口展开，包括 1D、3D、1W、1M、1Y、10Y 等窗口，以及 SiteId、ForecastId、day-of-week、time、day-off 等分组。
5. 代码为训练集设计了 random lag，用来模拟测试时只能看到带滞后的历史；同时为验证段生成 `noval` 聚合，避免验证 period 后面的真实能耗进入特征。
6. `lgb.py` 使用自定义 NWRMSE metric，按 ForecastId 拆分训练和验证，并把被选中验证的 ForecastId 后段作为 validation。
7. 代码中还在验证集替换 `_noval` 特征，显式排除验证 period 内不可用的信息。
8. 最终模型使用 LightGBM、特征开关、不同时间粒度的参数和 blending / retraining 流程。
9. winners 访谈中，第二名也强调用 time split 训练和验证以避免 leakage；第三名则把问题当作监督回归，但在单 building / ForecastId 层面确保只使用过去数据。

| --- | --- | --- |

1. **指标结构拆解规则**：不能只记录“指标名称”，必须拆解它如何加权、归一化、聚合、惩罚错误，并说明 Plan 是否与指标结构一致。
2. **预测时可用信息合约**：凡是预测、推荐、调度、干预类项目，必须按目标时间点列出当时可见和不可见的信息；所有特征、访谈证据和行动建议都要经过这个合约过滤。
3. **复现成本闸门**：公开答案或候选路线如果需要高算力、长时间、特殊数据、组织权限或复杂协作，Plan 必须提供低成本代理闭环，不能直接把高成本路线当成可执行方案。

题面事实：

1. 任务是从航拍/无人机影像中做建筑 footprint 语义分割，逐像素判断是否为建筑。
2. 数据来自非洲多个城市和区域；problem description 写明是 10 个城市和区域，winners blog 进一步提到 12 个城市和区域、70 多万建筑。
3. 训练影像是 Cloud Optimized GeoTIFF，包含 red、green、blue、alpha 四个 band；alpha 可用于 NoData mask。
4. 训练标签来自 GeoJSON building footprints，除 Zanzibar 外主要来自 OpenStreetMap；训练数据按标签质量分为 tier 1 和 tier 2，tier 1 更完整。
5. `train_metadata.csv` 包含 region 和 tier；test set 不提供 region。
6. test set 有 11,481 个 1024 x 1024 COG chips，来自训练集中没有出现的 scenes；部分 test regions 在训练中出现，部分没有。
7. test chips 的正确 georeferences 被移除；test labels 对参赛者不可见，且质量接近 tier 1。
8. 正式指标是 pixel-based Jaccard index / IoU，越高越好。
9. 提交格式是每个 test chip 一个单 band 1024 x 1024 TIFF mask，像素值为 True/False 或 0/1。
10. Segmentation track 参赛者必须至少向 Responsible AI track 提交一次，才有资格获得 segmentation performance prizes。
11. Responsible AI track 要求参赛者从灾害风险管理场景出发，讨论框架、潜在伤害、伦理关注点和缓解方式。
12. 题面举出的潜在风险包括：在脆弱、冲突、暴力地区加重隐私与安全风险；因数据产品复杂而限制公众参与；削弱专家判断；未充分沟通方法、结果和不确定性导致误用。

### 5.2 按协议进行的前置治理推演

#### 问题层级

| 层级 | 判断 |
| --- | --- |
| 世界问题 | 快速城市化和灾害风险管理需要频繁、准确、细粒度地更新建筑和基础设施地图，尤其是在数据更新困难、城市变化快的非洲城市 |
| 经验规格 | 竞赛规格是像素级 Jaccard；Responsible AI 规格是是否识别和缓解下游伦理风险；现实规格是能否帮助本地政府、社区和制图者更好地更新开放地图 |
| 操作问题 | 用多区域、高分辨率、标签质量不均的影像和 GeoJSON 训练建筑分割模型，并在未知 test scenes / regions 上泛化 |
| 表达问题 | 需要同时说明模型分割能力、空间泛化、标签噪声处理、可复现性，以及在 DRM 下游场景中如何避免误用和伤害 |

#### 资源边界

| 字段 | P04 判断 |
| --- | --- |
| allowed_basis | 竞赛提供的 train tier 1 / tier 2 影像、GeoJSON labels、metadata、STAC、test chips、Responsible AI instructions、rules |

#### 验证切分、场景异质性与目标形态

```yaml
validation_semantics: 空间域迁移；test scenes 不在训练集中，部分 test regions 也不在训练中
segment_or_context: city / region / scene / sensor / spatial resolution / tier / label quality / urban density / building size / NoData / chip boundary
target_form: 1024x1024 binary segmentation mask
metric_structure: pixel-level Jaccard = intersection / union；可能掩盖小建筑、稀有地区、边界质量和下游属性错误
qualitative_gate: Responsible AI framework / harm identification / mitigation / uncertainty communication
chosen_strategy: 先用 tier 1 + U-Net/FPN baseline 复现 Jaccard，再做空间分组验证、标签噪声诊断和 Responsible AI 风险表
what_this_validation_can_prove: 模型在若干 held-out scenes/regions 上的像素重叠能力
what_this_validation_cannot_prove: 不能证明下游风险评估公平、社区参与充分、地图可被本地机构安全采用
```

#### 发散方向

1. **tier 1 强基线路线**：先只用更完整的 tier 1 labels，训练 U-Net/FPN，复现 Jaccard。优点是稳；风险是样本和场景覆盖不足。
2. **标签噪声路线**：比较 tier 1 / tier 2，做 label quality audit、噪声鲁棒 loss 或伪标签。优点是贴近题面难点；风险是把错误标签放大。
3. **空间泛化路线**：按 region / scene 留出验证，不只随机 chip split。优点是模拟 test；风险是训练样本进一步减少。
4. **上下文与边界路线**：mosaic、overlap tile、TTA，避免 chip 边缘建筑被截断。优点是改善建筑边界；风险是算力和实现成本高。
5. **Responsible AI 路线**：围绕 local mappers、居民、政府、灾害响应者和开放数据使用者，识别误用、偏差、隐私和不确定性沟通风险。优点是服务公共意义；风险是容易写成空泛伦理口号。
6. **部署工具路线**：把模型输出接入人工制图校验，而不是替代本地专业判断。优点是更贴近 OpenDRI / OSM；风险是超出竞赛代码范围。

#### 主链收束

> 在禁止外部数据和 test labels 的边界下，先用 tier 1 数据建立可复现的 U-Net/FPN 分割基线，并用空间分组验证检验跨 scene / region 泛化；随后处理标签噪声、边界上下文和高成本模型技巧，同时把输出限定为“辅助本地制图与风险沟通的候选建筑 mask”，而不是直接替代本地判断或灾害资源分配。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 数据审计 | COG / GeoJSON / metadata / STAC | region、tier、分辨率、NoData、建筑面积分布 | 统计 + 可视化抽样 | 如果 label 与影像错位，先不训练 |
| 指标复现 | mask truth / mask pred | Jaccard 计算器 | 与 scikit-learn micro Jaccard 对照 | 如果只能看像素准确率，不能比较方案 |
| 空间基线 | tier 1 images + labels | U-Net/FPN baseline | held-out scene / region Jaccard | 如果随机 chip split 虚高，不能进入部署叙事 |
| 标签噪声闭环 | tier 1 vs tier 2 sample | label quality map | 人工抽样 + 分层误差 | 如果 tier 2 造成下降，作为后续候选而非主线 |
| 下游风险表 | Responsible AI instructions | stakeholder / harm / mitigation table | 对照 framework / identification / mitigation rubric | 如果只讲“模型很有用”，不能视为 Responsible AI 合格 |

### 5.3 查阅答案后的真实做法

答案来源包括 Open Cities winners repo、第一名 README、winners blog 和 Responsible AI 一等奖报告。

公开答案的关键做法包括：

1. winners repo README 总结：第一名是多阶段 pipeline，先用 noisy labels 和强 augmentations 训练 U-Net ensemble，再用 stage 1 模型自动标注 test data 形成 pseudolabels，继续训练两轮。
2. 第一名 README 说明该方案基于 U-Net-like CNN，并用 Docker / Docker Compose / Nvidia-Docker 封装环境。
3. 第一名方案硬件要求很高：16GB 显存 GPU、256GB 空闲磁盘、64GB RAM；训练配置需要至少 16GB 显存。
4. Stage 0 将 test tiles 拼成大 scenes / mosaic，以提供更多上下文，减少边界处建筑判断错误。
5. Stage 1 用 `train_tier_1`，重采样到 0.1m/pixel，从 GeoJSON 生成 raster masks，切成 1024 x 1024 tiles，训练 10 个 U-Net 模型。
6. Stage 1 使用 EfficientNet-B1 和 SE-ResNeXt 编码器、5 folds、hard augmentations、TTA，并对预测取平均后用 0.5 阈值。
7. Stage 2 / Stage 3 使用上一阶段预测作为 pseudolabels，继续训练多模型 ensemble。
8. winners blog 中第一名强调：强 augmentations 改善泛化；mosaic 可以避免模型只看到建筑边缘而误判。
9. 第二名强调 rare tiles oversampling、训练时尽量保留 inference 条件、多个模型平均集成。
10. 第三名使用 FPN + EfficientNet、Focal loss / Dice loss、noisy student、negative mining 和 TTA，但也承认有些技巧更适合 competition mode。
11. Responsible AI 一等奖报告讨论“航拍图像模型如何含有偏见”，强调标签偏差、地理/灾害样本覆盖、误报误漏对不同社区的影响、地方知识和人工监督的重要性，以及不确定性沟通。

## 6. P05：STAC Overflow: Map Floodwater from Radar Imagery

### 6.1 题面隔离

只读题面时得到的 current_basis：

| id | source | type | role | current_basis | reason |
| --- | --- | --- | --- | --- | --- |
| P05-M1 | DrivenData overview | problem | authoritative input | yes | 明确世界问题：近实时洪水范围制图可支持预警、风险评估和救援投放 |
| P05-M2 | Problem description | data schema / metric | authoritative input | yes | 明确 Sentinel-1 SAR、VV/VH、训练集、test runtime、Jaccard 和 mask 输出 |
| P05-M3 | Code submission format | execution contract | authoritative input | yes | 明确代码执行挑战、无网络、资源限制、目录结构、输出格式和 8 小时限制 |
| P05-M4 | Rules | rules | authoritative input | yes | 明确禁止未允许外部数据、test sample 独立处理和赢家复现要求 |
| P05-M5 | floodwater-runtime repo | runtime spec | authoritative input | yes | 明确 Docker runtime、本地测试、metric script 和环境包 |
| P05-M6 | winners repo / winners writeups | answer | answer-only | no_before_answer | 题面阶段禁止读取；赛后主页出现的赢家摘要也不进入审题依据 |

题面事实：

1. 任务是用 Sentinel-1 global SAR imagery 逐像素检测 floodwater。
2. 训练数据是 2016-2020 年不同地区的 SAR satellite imagery，由 Cloud to Street 标注。
3. 每个 chip 有两幅 512 x 512 GeoTIFF：`_vv` 和 `_vh` polarization bands；一个 chip 对应一个 512 x 512 label mask。
4. 训练集包含 542 个 chips、1084 张影像，来自 13 个 flood events。
5. 像素值以 dB 表示，0.0 表示 missing data；评分只在 valid input pixels 上进行，missing pixels 会被排除。
6. 可选补充数据包括 NASADEM elevation 和 JRC global surface water layers：seasonality、occurrence、change、recurrence、transitions、extent。
7. 外部数据不允许；但免费公开的预训练 CV 模型可以使用，且题面明确提供的 NASADEM/JRC 补充数据可通过 STAC API 或 runtime 静态资源使用。
8. test images 只在 runtime container 中可访问，挂载在 `data/test_features`；test images 不带 georeference。
9. 推理时，与 test chip 对应的 NASADEM/JRC 补充数据已作为 `data/nasadem` 和 `data/jrc_*` 静态资源提供，无需网络。
10. 这是 code execution challenge：不能直接提交预测标签，而要提交包含 `main.py` 和模型资产的 `submission.zip`，由平台在容器中执行。
11. 提交脚本必须读取 `data/test_features`，向 `submission` 文件夹写出每个 chip 一个 512x512 single-band GeoTIFF，像素值为 1/0。
12. runtime 限制包括：无网络、8 小时内完成、单 GPU、5 vCPU、48GB RAM、Tesla K80 12GB、无 root 权限。
13. 规则要求量化评估时每个 test sample 独立处理，不得使用其他 test cases 的信息，除非竞赛网站另有说明。
14. 赢家代码必须说明构建和运行资源，并能复现 winning results，也能在新数据样本上有无 retraining 地生成预测。

### 6.2 按协议进行的前置治理推演

#### 问题层级

| 层级 | 判断 |
| --- | --- |
| 世界问题 | 洪水是高频、高成本自然灾害；近实时 flood extent 能帮助政府和人道组织进行预警、风险评估和救援投放 |
| 经验规格 | 竞赛规格是 valid pixels 上的 binary Jaccard/IoU；执行规格是容器内无网络、限时、限资源、chip-level inference；现实规格是洪水范围是否及时、可靠、可被救援系统承接 |
| 操作问题 | 在 runtime 和 sample-independence 约束下，使用 VV/VH SAR、NASADEM、JRC 等授权数据生成每个 test chip 的二值水体 mask |
| 表达问题 | 需要解释 SAR 的物理机制、missing-data mask、补充数据边界、容器可复现性和下游救援误用风险 |

#### 资源边界

| 字段 | P05 判断 |
| --- | --- |
| allowed_basis | 训练 SAR VV/VH、labels、metadata、题面允许的 NASADEM/JRC、免费开放预训练 CV 模型、runtime repo、metric script |
| forbidden_basis | 未授权外部数据、test labels、test georeference、网络访问、跨 test sample 信息泄漏、硬编码 test chip、依赖平台外资源 |
| gray_zone | 公开预训练模型、Planetary Computer Hub、补充数据下载流程；需要区分 training-time 使用和 inference-time 静态资源 |

#### 验证切分、场景异质性与目标形态

```yaml
validation_semantics: 按 flood event / geography 留出，模拟新洪水和新地区；不能只随机拆 pixels
segment_or_context: flood_id / country / date / SAR polarization / vegetation / elevation / permanent water / missing-data pixels / event severity
target_form: 512x512 binary floodwater mask
metric_structure: valid pixels 上的 binary Jaccard；missing pixels 排除，IoU 对小水体、边界和 false positives/false negatives 的权衡敏感
evidence_or_label_quality: label mask、missing-data mask、SAR speckle/noise、永久水体与临时洪水混淆
execution_contract: submission.zip 顶层 main.py；无网络；8 小时；5 vCPU/48GB RAM/Tesla K80；读 data/test_features 与静态补充数据；写 submission/*.tif
chosen_strategy: 先做物理启发阈值/简单像素 baseline，再做授权补充数据 + U-Net 或像素分类模型，并先通过 runtime 本地测试
what_this_validation_can_prove: 模型能否在模拟新 flood/geography 的 chip-level 输入上生成有效 mask 并通过容器执行
what_this_validation_cannot_prove: 不能证明现实近实时系统的数据延迟、人工复核、救援资源投放和误判成本已经解决
```

#### 发散方向

1. **SAR 物理基线路线**：利用水面在 SAR 中常呈暗像元、VV/VH 组合、missing mask 和简单阈值做 baseline。优点是低成本、可解释；风险是风浪、植被、地形和城市环境会改变后向散射。
2. **授权补充数据路线**：加入 NASADEM 与 JRC permanent water layers，区分地形低洼、永久水体和临时洪水。优点是贴近水文机制；风险是把永久水误当 flood 或引入静态数据偏差。
3. **语义分割路线**：U-Net / U-Net++ / EfficientNet backbone，对 512x512 VV/VH/DEM 多通道图像做分割。优点是捕捉空间结构；风险是训练样本少、泛化不稳、算力高。
4. **像素分类路线**：把 VV、VH、DEM、JRC layers 展平成 pixel-level 表格，用 CatBoost/GBDT 做二分类。优点是简单、资源可控、对小数据友好；风险是缺少空间上下文。
5. **事件分组验证路线**：按 flood event / geography 留出，而不是随机 pixels。优点是模拟真实新灾害；风险是验证集小、分数波动大。
6. **runtime-first 路线**：先用 dummy 或轻模型通过 `submission.zip`、无网络、输出 TIFF、格式测试和 8 小时限制。优点是降低执行失败风险；风险是早期精度低。
7. **下游救援风险路线**：标注 false negative 与 false positive 的不同代价，设计人工复核和不确定性输出。优点是服务真实效用；风险是竞赛提交只能交二值 mask。

#### 主链收束

> 在无网络、限时、chip 独立推理的 runtime 契约下，先建立可解释的 SAR + missing mask + 简单阈值/像素模型 baseline，并复现 valid-pixel Jaccard；再加入题面授权的 NASADEM/JRC 补充数据和轻量 U-Net 或 CatBoost 集成，按 flood event / geography 验证泛化，最后确保 `main.py` 在官方容器中能独立读取每个 chip 并输出合格 GeoTIFF。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 数据审计 | VV/VH GeoTIFF、labels、metadata | flood_id、date、country、missing pixel、label 水体比例 | 统计 + 抽样可视化 | 如果 missing mask 没处理，不能算正式指标 |
| 指标复现 | label / pred / valid mask | binary Jaccard | runtime metric script 或 sklearn 对照 | 如果把 missing pixels 计入，停止 |
| 物理基线 | VV/VH + 简单阈值 | baseline masks | event-level holdout IoU | 如果低于全 0/全 1 或明显无效，回到数据审计 |
| 补充数据闭环 | NASADEM/JRC + VV/VH | feature set v1 | 对永久水/临时洪水错误分层 | 如果补充数据只提升随机 split，不写成泛化 |
| runtime 闭环 | `submission.zip` + dummy/轻模型 | 合格 `.tif` 输出 | floodwater-runtime 本地测试 | 如果无网络、路径、命名或时间失败，先修执行契约 |
| 下游风险表 | flood mask 错误案例 | false positive / false negative 风险 | 对照救援投放与预警场景 | 如果只报 IoU，不写成救援可用 |

## 7. P06：Mars Spectrometry: Detect Evidence for Past Habitability

### 7.1 题面隔离

只读题面时得到的 current_basis：

| id | source | type | role | current_basis | reason |
| --- | --- | --- | --- | --- | --- |
| P06-M1 | DrivenData overview / NASA page | problem context | authoritative input | yes | 明确世界问题：用质谱自动分析帮助科学家理解火星过去宜居性 |
| P06-M2 | Problem description | data schema / metric | authoritative input | yes | 明确 EGA mass spectrometry、样本级多标签、log loss、validation/test split |
| P06-M3 | Data notes / scientific notes | domain constraints | authoritative input | yes | 明确 abundance 只能样本内相对比较、峰形/峰值、背景离子、He carrier gas、m/z 含义 |
| P06-M4 | Competition rules | rules | authoritative input | yes | 明确外部数据、人工标注、test sample independent、赢家复现要求 |
| P06-M5 | winners repo / writeups | answer | answer-only | no_before_answer | 题面阶段禁止读取 |

题面事实：

1. 任务是从 evolved gas analysis mass spectrometry 数据中，检测 geological material samples 中若干 chemical compound families 是否存在。
2. 世界意义是支持 Mars exploration mission 中的科学分析，帮助理解火星过去宜居性的证据，而不是直接证明“火星存在生命”。
3. 观测单位是物理样本。每个样本的特征文件是长表 CSV，包含 `time`、`temp`、`m/z` 和 `abundance`。
4. 标签是多标签二分类，每个样本可能同时含有多个 compound families。
5. 提交文件对 validation 和 test 样本输出每个 label 的 0 到 1 置信度，正式指标是 aggregated log loss。
6. log loss 会惩罚过度自信的错误预测，因此输出不是普通标签，而是需要校准的概率或置信度。
7. 训练数据包含 commercial instrument 与 SAM testbed 两类仪器。二者单位、采样方式、m/z 范围和样本数量不同。
8. commercial abundance 单位与 SAM testbed 不同，abundance 应主要在同一样本内部相对比较，不能跨仪器直接比绝对值。
9. SAM testbed 样本更接近 Curiosity rover 上 Sample Analysis at Mars 的仪器环境，但训练样本更少，且部分 label classes 在训练中可能没有出现、最终 test 中可能出现。
10. 题面提供 unlabeled supplemental data，可用于 unsupervised 或 semi-supervised 方法；但 supplemental samples 的实验参数可能与主数据不同，例如 carrier gas 不同。
11. 外部数据只在公开、自由可获得、许可合适且在论坛共享时允许；训练数据可人工标注但必须随方案提交，test data 不允许人工标注。
12. test samples 必须独立处理，不得用其他 test cases 的信息或 test-set pseudo-labeling；赢家代码要能在新数据上自动生成预测。
13. 题面另设 SAM testbed bonus prize，说明除了主指标，还有科学家评审的重点子场景和应用潜力判断。

#### 验证切分、场景异质性与目标形态

```yaml
validation_semantics: train/validation/test split；validation labels 阶段性释放，final ranking 基于 test；还要单独观察 SAM testbed 子场景
segment_or_context: instrument_type, commercial vs sam_testbed, sample_id, label class, carrier_gas, m/z range, temperature window, background ions, supplemental vs primary data
target_form: 样本级 10 类 multi-label probability vector
metric_structure: unweighted aggregated log loss across observations and labels；类别稀有度不自动加权；过度自信错误会被重罚
evidence_or_label_quality: 小样本、多标签、部分 label class 未在训练出现、不同仪器单位和采样协议、背景/噪声/He carrier gas
prediction_time_contract: 给定单个新样本的 feature CSV 与 metadata，独立输出概率；不得依赖其他 test samples 或事后标签
official_supplementary_basis: unlabeled supplemental data 可用，但必须记录实验参数差异和分布警告
scientific_claim_boundary: 模型输出只能支持“某化合物家族存在概率”的分析，不能单独构成火星宜居性或生命迹象结论
chosen_strategy: 先做领域启发的 per-sample 归一化、m/z/temp 曲线统计、峰值/面积/宽度 baseline，再比较图像化谱图模型和表格特征模型，并对概率做校准
what_this_validation_can_prove: 模型能否在给定质谱样本上输出相对可靠的 compound-family 概率
what_this_validation_cannot_prove: 不能证明模型已经替代科学家判断，也不能证明火星过去宜居性的科学命题
```

#### 发散方向

1. **领域特征 baseline 路线**：按 m/z 和 temperature 对 abundance 曲线提取面积、峰值、峰宽、峰温、比值和背景扣除特征。优点是贴近科学 notes；风险是手工特征不能捕捉复杂谱图结构。
2. **谱图图像路线**：把 sample 表示为 temperature x m/z 的二维图像，用 CNN/Transformer 处理。优点是保留结构；风险是样本少、预处理选择影响大。
3. **序列模型路线**：把不同 m/z channel 上的温度序列输入 RNN/1D CNN/Transformer。优点是符合 EGA 加热过程；风险是 commercial 与 SAM testbed 采样方式不同。
4. **仪器分层路线**：对 commercial 与 SAM testbed 分层建模或分层验证，重点记录单位和采样协议差异。优点是尊重真实仪器差异；风险是 SAM testbed 样本少。
5. **概率校准路线**：用 log loss 驱动输出校准概率，避免对缺失/未见类别输出过度自信的 0。优点是匹配正式指标和科学不确定性；风险是验证集太小导致校准不稳。
6. **supplemental 半监督路线**：利用 unlabeled supplemental data 学习谱图表示或分布，但对 carrier gas 和实验参数差异做单独标注。优点是增加数据；风险是把不同实验协议混为一谈。
7. **科学解释路线**：将模型依据映射回 m/z、temperature peak 和 compound-family 线索，作为科学家复核材料。优点是服务真实使用；风险是解释不等于因果证明。

#### 主链收束

> 在只使用题面允许材料、且 test sample 独立预测的边界下，把每个 EGA 质谱样本转化为尊重仪器差异和样本内相对 abundance 的结构表示，先用领域特征与校准概率 baseline 复现 aggregated log loss，再用谱图图像/序列模型和 supplemental data 增强，并始终把模型输出限定为 compound-family probability，而不是火星宜居性结论本身。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 数据审计 | metadata、sample CSV、labels | instrument_type、label prevalence、m/z/temp 范围、abundance 尺度、supplemental 参数 | 统计 + 抽样谱图 | 如果跨仪器直接比较 abundance 绝对值，停止 |

### 7.3 查阅答案后的真实做法

答案来源包括 Mars Spectrometry winners repo、第一名 README、二三名 README 和部分赢家代码。

4. 第一名按 `instrument_type` 做 stratified folds，并检查 SAM testbed 在各 fold 的分布，说明仪器类型不是普通字段，而是验证语义的一部分。
5. 第二名 README 显示，训练需要约 15 小时 LGB、约 100 小时 NN，并使用 supplemental_features；其代码提取 means、totals、raw_peak、peak_temp、peak_to_mean、width_at_half、jitter、prevalence 等谱图统计和峰特征。
6. 第三名将多标签问题改写为 `(sample_features, compound_name)` 的二分类问题，先用 LightGBM ensemble 得到初始预测，再把这些预测和 top features 输入 31-fold CatBoost meta model；其 README 明确说 compound 之间的共现信息有帮助。
7. 第三名预处理代码删除 fractional m/z、m/z >= 100 和 carrier gas `m/z = 4`，按样本归一化 abundance，并将每个 m/z 的 abundance/temp 序列重采样为固定长度特征；代码注释强调特征工程按 sample_id 进行，以避免泄漏。
8. NASA/DrivenData 结果页强调 winning solution 取得 0.092 log loss 和 0.95 average precision，并且有望帮助分析火星数据、影响未来 planetary mission instruments 的原位分析设计。

候选规则：

1. **科学/研究结论边界规则**：如果模型、调研或方案只测量了代理目标，必须写清它能支持什么结论、不能支持什么结论；不能把“化合物家族概率”“问卷态度”“课堂反馈”直接写成宏大科学或社会结论。

5. **重点子场景/第二评价规则**：如果项目存在 bonus prize、评委重点、关键人群、关键学校、关键场景或专家质性评审，必须单列 priority subset 和 secondary evaluation。

## 8. P07：NASA Airathon: Predict Air Quality

### 8.1 题面隔离

只读题面时得到的 current_basis：

| id | source | type | role | current_basis | reason |
| --- | --- | --- | --- | --- | --- |
| P07-M1 | NASA Airathon group overview | problem context | authoritative input | yes | 明确世界问题：空气污染暴露与健康风险，需要高分辨率日尺度空气质量信息 |
| P07-M2 | PM2.5 problem description | data schema / metric / rules | authoritative input | yes | 明确 PM2.5 track 的目标、批准数据、标签、提交格式、R2/RMSE 和外部数据规则 |
| P07-M3 | NO2 problem description | data schema / metric / rules | authoritative input | yes | 明确 NO2 track 的目标、批准数据、标签、提交格式、R2/RMSE 和外部数据规则 |
| P07-M4 | winners repo / winners blog | answer | answer-only | no_before_answer | 题面阶段禁止读取；竞赛页顶部出现的 winner 信息不用于审题推演 |

题面事实：

1. NASA Airathon 分为两个 track：Particulate Track 预测 PM2.5，Trace Gas Track 预测 NO2。
2. 任务是在 Los Angeles (South Coast Air Basin)、Delhi、Taipei 三个城市的 5km x 5km grid cells 上，生成每日 surface-level 污染物浓度估计。
3. 目标单位是微克每立方米，标签来自 reference-grade ground monitors。每条记录由 `datetime` 和 `grid_id` 唯一确定。
4. 训练与测试是时空混合结构：PM2.5 训练期为 2018-02 至 2020-12，测试期包括 2017-01 至 2018-01 和 2021-01 至 2021-08；NO2 训练期为 2019-01 至 2020-11，测试期包括 2018-08 至 2018-12 和 2020-11 至 2021-08。
5. approved input 包括卫星数据和 ancillary data。题面要求至少使用一个 approved satellite source。
6. PM2.5 的核心卫星变量是 AOD/AOT，主要来源包括 MAIAC 和 MISR；NO2 的核心卫星变量是 NO2 vertical column density，主要来源包括 OMI 和 TROPOMI。
7. ancillary data 包括 NASADEM、GFS、ECMWF IFS、GEOS-CF 等，但只能使用题面列出的 approved access location。
8. 题面明确提醒：数据有不同的时空分辨率、格式、覆盖日期和地理范围，仪器错误会导致缺失或错误值。
9. 外部数据不允许，除非通过官方流程被批准；候选外部数据必须公开免费、可靠运行、不能使用 reference-grade surface monitor 数据，并能证明相对现有 approved sources 有新增价值。
10. 预测时只能使用 estimation day 当天及之前可见的数据，训练数据除外；禁止使用 estimation day 之后的未来数据。
11. 与 P05/P06 不同，题面明确允许一个 test grid cell 的预测使用推理时点可见的其他 test samples 特征和预测，不要求每个 test sample 完全独立处理。
12. 赢家不仅要提交预测，还要提交可审计代码：模型要能输入一个新日期，在不重新训练的情况下，为三个城市同一天所有相关 grid cells 输出预测；一小时内在单 GPU 节点完成；并在 out-of-sample time period 上通过复核。
13. 正式排名指标是 R squared，leaderboard 还报告 RMSE；只有 R2 决定正式排名和 prize eligibility。

### 8.2 按协议进行的前置治理推演

#### 问题层级

| 层级 | 判断 |
| --- | --- |
| 世界问题 | 空气污染是重要公共健康风险，但地面监测站昂贵且覆盖有限，许多人缺少日常、局地的空气质量信息 |
| 经验规格 | 竞赛规格是在 withheld dates / grid cells 上提高 R2，并报告 RMSE；真实规格是模型能否在单日、多城市、高缺失卫星数据下提供可信的污染暴露估计 |
| 操作问题 | 将不同分辨率、不同时间戳、不同格式的卫星/气象/地形数据映射到 5km grid-date，并严格保证只使用推理时点可见信息 |
| 表达问题 | 必须说明模型估计的是 surface-level pollutant concentration 的代理值，不等于每个人真实暴露剂量；还要说明缺失、时间延迟、城市差异和不确定性 |

#### 资源边界

| 字段 | P07 判断 |
| --- | --- |
| allowed_basis | 训练标签、grid metadata、satellite metadata、approved satellite sources、approved ancillary data、题面规则、官方 benchmark 思路 |
| forbidden_basis | 未批准外部数据、未来日期特征、由 reference-grade monitors 派生的外部模型或数据、test labels、私下共享数据 |
| official_supplementary_basis | MAIAC/MISR/OMI/TROPOMI、NASADEM、GFS、IFS、GEOS-CF 等；必须使用指定 approved access location |
| gray_zone | 其他遥感、天气、道路、排放、人口、低成本传感器数据；需要官方审批且不得含 reference-grade monitor 信息 |
| distribution_warning | 三城市污染水平、监测覆盖、气候、卫星覆盖和历史数据量差异大；测试期还包括训练期之前和之后 |
| boundary_reason | 公共健康、竞赛公平、未来信息泄漏、地面监测标签污染、可部署数据访问 |

#### 验证切分、场景异质性与目标形态

```yaml
validation_semantics: 时间外推 + 城市/网格空间泛化；还要模拟单个新日期批量推理
segment_or_context: pollutant track, city/location, grid_id, date, season, timezone, satellite product, data source latency, missingness, monitor coverage
target_form: grid-date level continuous pollutant concentration value
metric_structure: official R2 ranking + RMSE reported；R2 相对总体均值，容易受城市均值、方差和极端污染事件影响
prediction_time_contract:
  actor: 已训练模型或空气质量信息服务
  action_time: estimation day / single new day
  visible_information: 当天及之前已生成且按 approved access location 可获得的卫星/气象/地形数据、训练标签
  invisible_information: estimation day 之后的数据、test labels、未批准外部监测数据
case_context_contract:
  unit_of_prediction: grid_id x datetime
  same_batch_visible_information: 同一新日期、同一可见时点内其他 grid cells 的 approved features / predictions 可用
  forbidden_context: 未来日期、隐藏标签、未批准外部数据
execution_contract: 输入新日期，下载/处理相关数据，一小时内输出三个城市同日所有 grid cells 的 prediction CSV
domain_preprocessing_assumptions: 卫星栅格到 grid polygon 的聚合、缺失值插补、时间窗口、时区转换、forecast valid_time 对齐
claim_ladder: 卫星/气象数据估计地表污染浓度，不等于个人暴露、因果归因或健康结果预测
chosen_strategy: 先做城市均值和 grid/date baseline，再构建 as-of-time 安全的 satellite + weather + elevation 特征，按时间/城市分层验证
what_this_validation_can_prove: 模型在相似数据可得性和新日期批量推理条件下估计污染浓度的能力
what_this_validation_cannot_prove: 不能证明所有城市/低监测覆盖地区可用，也不能直接证明个人健康风险降低
```

#### 发散方向

1. **城市均值/季节 baseline 路线**：先用 city mean、grid mean、month/day features、上一可见窗口做 baseline。优点是能识别城市间差异的基本贡献；风险是不能捕捉城市内空间变化。
2. **卫星特征路线**：按 grid polygon 从 MAIAC/MISR/OMI/TROPOMI 提取 mean、variance、percentile、coverage、quality flags。优点是贴近题面核心；风险是缺失多、单位和空间分辨率不同。
3. **气象/地形补充路线**：加入 GFS/IFS/GEOS-CF/NASADEM，解释污染扩散、边界层、风、降水和地形影响。优点是符合空气污染机理；风险是 forecast valid_time 和生成时间容易泄漏。
4. **城市分治路线**：三城市污染机制和数据覆盖差异大，可比较全局模型、location-specific 模型和 ensemble。优点是尊重异质性；风险是每城样本少。
5. **缺失建模路线**：将 satellite missingness 当作信号，比较 grid-wise mean、time interpolation、last available、spatial neighbor imputation。优点是贴近题面难点；风险是插补如果使用未来会泄漏。
6. **批量上下文路线**：利用题面允许的同一推理时点内跨 grid context，例如 spatial smoothing / neighbor features / same-day city distribution。优点是利用空间结构；风险是越过题面允许范围。
7. **真实使用路线**：把输出设计成城市/日期污染热力信息，并附不确定性与覆盖质量。优点是连接公共健康；风险是竞赛提交只有单值，无法表达置信度。

#### 主链收束

> 在只使用 approved sources 且严格遵守 as-of-time 可见性边界的前提下，把多源卫星、气象和地形数据按 grid-date 对齐为缺失可解释的特征，先复现 city/grid/date baseline 与 R2/RMSE，再比较全局模型、城市分模型和同日批量上下文模型，最后确保方案能在一个新日期内为三个城市所有 grid cells 输出可审计预测。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 数据可见性账本 | labels、satellite metadata、grid metadata、approved data docs | 每个数据源的 observation time、generation time、access location、valid_time | 人工审计若干 grid-date | 如果用到未来 granule 或未批准入口，停止 |
| 指标复现 | train labels + dummy / city mean predictions | R2 与 RMSE | 本地实现对照 leaderboard 公式 | 如果只看 RMSE，不足以模拟正式排名 |
| 基线 | city mean、grid mean、month、day | baseline submission | 时间切分 + 城市分层误差 | 如果不能超过城市均值，先做数据对齐而非复杂模型 |
| 卫星/气象特征 | AOD/NO2 VCD/GFS/NASADEM | grid-date feature table | 缺失率、coverage、时区/valid_time 检查 | 如果插补偷用未来值，停止 |
| 城市/网格验证 | location、grid_id、date | per-city R2/RMSE | location-specific vs global model 对比 | 如果总分掩盖某城失效，不能声称公共可用 |
| 批量上下文闭环 | 单日所有 grid cells | same-day batch inference output | 明确同日可共享信息与不可共享信息 | 如果跨未来日期或标签传播，停止 |
| 执行契约闭环 | 新日期、approved access credentials、trained model | 1 小时内 prediction CSV | dry-run / small date window | 如果依赖不可获得账号或下载时间过长，回到低成本代理 |

### 8.3 查阅答案后的真实做法

答案来源包括 NASA Airathon winners repo、PM2.5/NO2 winners README、赢家博客和部分第一名/二三名 README。

公开答案的关键做法包括：

1. winners repo README 将目标定位为用 remote sensing 与 geospatial data 估计 PM2.5 和 NO2 日尺度污染水平，服务公众降低空气污染暴露。
2. PM2.5 第一名使用 MAIAC、MISR、GFS、NASADEM，提取每个 grid cell polygon 内各 band 的 mean 和 variance，做 grid-wise mean imputation 和 temporal difference features，再训练 XGBoost/CatBoost/LGBM，并融合 location-wise 与 all-location 两条 pipeline。
3. PM2.5 第一名特别强调对每个 observation 选择 `satellite_time_end < observation_time_end` 的最新文件，说明真正的工程核心是 as-of-time 对齐。
4. PM2.5 winners README 显示，前三名均大量使用 approved satellite + GFS；第二名用 45 个 LGBM 模型，第三名用 MAIAC + GFS、线性插补、随机森林/GBR ensemble，并用文献选择气象变量。
5. NO2 第一名使用 LGBM + ElasticNet 和神经网络 ensemble，采用 purged KFold cross-validation with a 30-day gap，fold / parameter rotation，并按 location 独立训练模型；还用 grid point 周围同心圆提取空间特征。
6. NO2 第一名 README 详细说明了 UTC 到 local time 的转换、GFS/IFS/Assim/TROPOMI 等不同数据源的时间对齐，并用 trailing data 和 moving average 处理预测窗口。
7. NO2 第二名 README 明确用 `obs_datetime_end` 过滤：Satellite、GFS 和 GEOS 数据只取 observation end 之前可获得的数据，并解释 GEOS hindcast 的 data_generation_datetime。
8. winners blog 总结了共同模式：所有六位赢家都用 tree-based models；imputation 是关键预处理；多数赢家选择 location-specific models 并与 overall model 或 ensemble 结合。
9. winners blog 还提供了结果背景：PM2.5 top model private R2 约 0.81，NO2 top model private R2 约 0.48；简单城市均值 baseline 在 PM2.5 上也能取得较强分数，说明城市均值/分布差异是必须显式建模的 baseline。

候选规则：

1. **可见信息账本规则**：预测、调研和现场行动都要记录 evidence/data 的生成时间、可访问时间、时区、延迟和权限；“当天的数据”不自动等于“当天行动时可用的数据”。
2. **同批次上下文规则**：不要默认所有样本必须独立，也不要默认可以互相借用。必须记录 unit_of_prediction、same_batch_visible_information、cross_case_allowed、cross_case_forbidden。

4. **批准入口规则**：官方允许数据不仅是“这个数据源可以用”，还包括从哪里访问、是否需要账号、是否有生成延迟、是否污染标签、是否能在推理/现场阶段复现。

## 9. P08：Snowcast Showdown / Water Supply Forecast Rodeo

### 9.1 题面隔离

只读题面时得到的 current_basis：

| id | source | type | role | current_basis | reason |
| --- | --- | --- | --- | --- | --- |
| P08-M1 | Water Supply Forecast Rodeo overview | problem context | authoritative input | yes | 明确世界问题：美国西部水资源管理依赖准确季节水量预报 |
| P08-M2 | Forecast Stage problem description | data schema / metric / execution | authoritative input | yes | 明确 26 个站点、0.10/0.50/0.90 分位数、issue dates、quantile loss、container inference |
| P08-M3 | Forecast Stage approved data page | resource boundary | authoritative input | yes | 明确只能使用 approved data sources、mounted data 与 Direct API Access |
| P08-M4 | Explainability and Communication Bonus Track | communication task / judging rubric | authoritative input | yes | 明确 forecast summaries、解释指标、面向水资源管理者的沟通产物 |
| P08-M5 | winners repo / winners blogs | answer | answer-only | no_before_answer | 题面阶段禁止读取 |

题面事实：

1. 任务是为美国西部 26 个 hydrologic sites 预测 seasonal naturalized cumulative streamflow volume，单位是 thousand acre-feet (KAF)。
2. seasonal water supply forecast 关注某个 forecast period 的累计径流量，多数站点 season 为 April-July。
3. Forecast Stage 需要在 2024 season 的 28 个 issue dates 上生成预测：1 月到 7 月每月 1、8、15、22 日。
4. 输出不是单点预测，而是 0.10、0.50、0.90 三个分位数；0.10/0.90 构成中心 80% 预测区间。
5. 题面明确提醒：水文业务中常用 probability of exceedance，它和 quantile notation 方向相反；不能把 90% exceedance 误解成 0.90 quantile。
6. 有些 issue dates 会落在 forecast season 内，已过去月份的 naturalized flow 可以作为已知部分，但目标仍是 total seasonal water supply，不是只预测 residual。
7. 训练标签来自 NRCS 和 River Forecast Centers 的 naturalized streamflow；Forecast Stage ground truth 要等 2024 season 结束后才可得，实际用 2024-08-19 finalized data 评分。
8. Forecast Stage 是代码执行挑战：提交冻结模型和预测代码，DrivenData 在 evaluation period 自动按 scheduled issue dates 运行，参赛者只在 code jobs fail 时修复。
9. 预测时必须只使用 issue date 之前的数据。例如 2024-03-15 forecast 只能使用 2024-03-14 或更早的 feature data。
10. 代码可能在 issue date 之后才运行，但必须产生与 issue date 当天运行相同的 forecast；因此必须显式按 issue_date 截断数据。
11. 默认 test samples 要独立处理，不得利用跨 test samples 的信息训练、伪标签或无监督学习；模型无需把 water years 在 Forecast Stage 中独立处理，但 Final Stage 的 cross-validation 又会关心水文年重叠。
12. approved features 包括 antecedent flow、snowpack、meteorology、seasonal forecasts、teleconnection indices 等；Forecast Stage 运行时通常使用 mounted data，只有 Direct API Access Approved 的数据源可在运行时下载。
13. primary metric 是三分位数 Averaged Mean Quantile Loss；secondary metric 是 interval coverage，理想 0.10-0.90 区间 coverage 约为 0.8，但不直接决定排名。
14. Explainability and Communication Bonus Track 要求提交四个 forecast summaries，面向 operational water resource managers，说明 forecast、uncertainty、历史背景、关键影响因子和 forecast changes；评分维度包括 usefulness、clarity、rigor、innovation。

#### 验证切分、场景异质性与目标形态

```yaml
validation_semantics: Hindcast historical holdout + Forecast Stage live 2024 rolling evaluation + Final Stage 20-year leave-one-out CV
segment_or_context: site_id, basin, RFC, issue_date, month, water year, forecast season, known months, snowpack regime, data source availability
target_form: site x issue_date 的 0.10/0.50/0.90 total seasonal streamflow quantile forecast
metric_structure: 三个分位数 mean quantile loss 平均；0.5 分位数损失等价 MAE；interval coverage 为校准诊断但非排名指标
prediction_time_contract:
  actor: frozen code submission / operational forecast model
  action_time: scheduled issue date
  visible_information: issue_date 前一天及更早的 approved feature data、历史训练标签、已过去月份 naturalized flow
  invisible_information: issue_date 当天之后的数据、2024 final ground truth、未批准外部数据、跨 test sample 学习
  data_availability_ledger:
    generated_at: 各数据源发布时间或 runtime mounted data 版本
    accessible_at: issue_date 运行时是否可得
    as_of_replay_rule: 即使代码晚于 issue_date 运行，也必须只用 issue_date 前可见数据
case_context_contract:
  unit_of_prediction_or_action: site_id x issue_date
  same_batch_visible_information: submission format 中同一 issue_date 待预测站点元数据可见
  cross_case_forbidden: 用其他 test forecasts/hidden outcomes 训练、伪标签或调整模型
execution_contract: 冻结模型 + predict function + container runtime + scheduled issue-date runs + failed job repair
secondary_evaluation_or_priority_subset: Forecast Stage、Final Stage、Hindcast Stage、Explainability Bonus、regional/site/month 误差
domain_preprocessing_assumptions: residual vs total 转换、水文年、已知月份流量累加、站点附近 snow stations 聚合、missing source handling
claim_ladder: 分位数 forecast 支持水资源调度判断，不等于确定水量或政策决策自动化
chosen_strategy: 先做 climatology / antecedent flow / snowpack baseline，再做 as-of-date 安全的分位数模型，并单独评估 interval coverage 和 site/month 分层表现
what_this_validation_can_prove: 模型能否在滚动 issue dates 上生成校准的 seasonal streamflow quantiles
what_this_validation_cannot_prove: 不能直接证明管理决策最优，也不能证明极端未来气候下可用
```

#### 发散方向

1. **气候常年值 baseline 路线**：用 site/month historical quantiles、seasonal climatology、forecast issue month 建立 baseline。优点是低风险；风险是不能利用当年积雪与气象异常。
2. **已知流量 + residual 路线**：对季内 issue dates，把已过去月份 naturalized flow 作为已知部分，再预测剩余季节流量并加回 total。优点是符合题面目标；风险是 residual/total 搞错会系统偏差。
3. **雪水当量/降水路线**：用 SNOTEL/CDEC/SWANN snowpack、cumulative precipitation、basin aggregation 描述水源储备。优点是物理相关；风险是站点选择和空间聚合容易过拟合。
4. **气象与遥相关路线**：加入 ACIS、PDSI、seasonal forecasts、ENSO/PDO/MJO 等 teleconnections。优点是补充大尺度气候信号；风险是小样本下信号弱。
5. **分位数模型路线**：分别训练 0.10/0.50/0.90 quantile models 或多输出模型，检查 quantile crossing 和 interval coverage。优点是贴合 metric；风险是区间不校准。
6. **流域分层路线**：按 RFC、basin、elevation、snow regime、issue month 分层验证。优点是避免平均分掩盖关键站点；风险是样本更少。
7. **解释沟通路线**：用 forecast plots、historical context、SWE/PPT/antecedent flow 条件和 SHAP/what-if 解释。优点是面向水资源管理者；风险是解释指标可能看似精美但不真实。

#### 主链收束

> 在冻结模型、approved data 和 as-of-issue-date 回放约束下，先建立 site/issue-month climatology 与已知流量 baseline，再用 snowpack、antecedent flow、气象和流域属性训练分位数模型，输出 total seasonal streamflow 的 0.10/0.50/0.90 预测，同时检查 interval coverage、quantile coherence、site/month 分层误差和面向水资源管理者的解释沟通。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 可见信息账本 | issue_date、approved data docs、runtime data | 每个数据源 as-of cutoff、mounted/API、可运行时间 | 抽样 issue dates 审计 | 如果晚运行时偷用 issue_date 之后数据，停止 |
| 指标复现 | labels + quantile predictions | MQL 与 coverage | sklearn pinball loss + coverage 手算对照 | 如果只算 RMSE/MAE，不进入建模 |
| 气候 baseline | historical site-year volume | site/month quantile baseline | LOOCV by water year | 如果不能打败 climatology，先修目标/切分 |
| 已知流量闭环 | monthly naturalized flow | known months + residual/total table | 季内 issue date 样例 | 如果把 residual 当 total，停止 |
| 分位数模型 | snowpack/precip/flow/climate features | 0.10/0.50/0.90 forecasts | MQL、coverage、quantile crossing 检查 | 如果 volume_10 > volume_50 或 coverage 极差，回到校准 |
| 分层验证 | site/RFC/month/water year | 分层误差表 | site/month/RFC 分组 | 如果总分掩盖关键流域失效，不能声称可用 |

### 10.1 题面隔离

只读题面时得到的 current_basis：

题面事实：

1. 世界问题是 giant kelp 对海洋生物多样性、碳循环、海岸生态和渔业有价值，但传统人工监测成本高、覆盖有限。
2. 竞赛聚焦 Falkland Islands 周边 kelp forest，要求从 Landsat satellite imagery 中 segment kelp canopy。
3. 输入是 350 x 350 pixel image tiles，包含 Landsat 的可见光、近红外、短波红外等多波段信息，并包含 DEM、cloud cover、land mask 等辅助层。
4. 输出是同尺寸二值 mask：每个像素判断是否为 kelp canopy。
5. 评分指标是 Dice coefficient，本质上奖励预测 mask 与 ground truth mask 的重叠；在 kelp 面积极小、背景极大的情形下，空白预测可能有误导性，需要单独检查 positive tiles 和边界质量。
6. 题面说明为了竞赛公平和保护测试集，数据中移除了 geospatial information；参赛者不能依赖真实经纬度或地点标签。
7. 训练样本和测试样本来自真实遥感观测，可能有云、陆地、海水反射、潮汐、季节、传感器质量和标注噪声差异。
8. 参赛者可以使用公开可得的预训练模型，但不能使用为本竞赛测试集或隐藏标签定制的材料。
9. 不能人工标注测试集、不能把测试影像拿去外部众包标注，也不能通过不被允许的外部数据恢复测试 tile 的真实位置。
10. 推理时每个 test image 应独立处理；不能利用其他 test images 做伪标签、无监督适配或跨测试样本学习。
11. 真实效用不是“分割得分高”本身，而是能否更可靠地生成 kelp canopy monitoring map，帮助生态研究和管理决策。
12. 代理边界很重要：Landsat 可见的是水面附近的 kelp canopy，不等于完整水下 kelp biomass、生态系统健康、长期恢复能力或政策成效。

#### 验证切分、场景异质性与目标形态

```yaml
validation_semantics: 跨 tile / 跨观测条件泛化；由于 geospatial information 被移除，不能假装验证了真实跨地点泛化
segment_or_context: kelp-present vs kelp-absent tile, cloud cover, land/water boundary, sensor band quality, coastline morphology, tide/season/sea state, label confidence
target_form: 350 x 350 binary semantic segmentation mask
metric_structure: Dice coefficient；需要处理 class imbalance、empty/near-empty masks、false positive kelp patches 和边界误差
ignored_or_valid_region: land/cloud/invalid pixels 是否参与评分必须按题面和数据说明验证；即使不计分，也应作为模型输入或后处理边界
evidence_or_label_quality: ground truth mask 可能来自遥感/专家处理，存在边界模糊、漏标、云遮挡和 temporal mismatch 风险
prediction_time_contract:
  actor: segmentation inference pipeline
  action_time: 对单个 unseen test tile 生成 mask
  visible_information: 当前 tile 的所有官方 bands / auxiliary layers、训练集、公开通用模型
  invisible_information: test labels、人工测试标注、其他 test tile 的标签或伪标签、真实地理坐标恢复结果
case_context_contract:
  unit_of_prediction_or_action: single 350 x 350 tile
  same_batch_visible_information: submission schema 可见，但不应从其他 test images 学习
  cross_case_forbidden: test-time pseudo-labeling, unsupervised adaptation across test tiles, manual review of hidden test masks
domain_preprocessing_assumptions: band scaling, cloud/land/water masking, spectral indices, DEM usage, threshold post-processing, morphological cleanup
claim_ladder: pixel mask -> visible kelp canopy map -> monitoring aid；不能直接推出 underwater biomass、生态健康或政策成效
chosen_strategy: 先做 spectral-index / simple CNN baseline，再做 U-Net/DeepLab 类分割模型；每步都按 Dice、positive tile recall、false positive rate 和可观测边界验证

what_this_validation_cannot_prove: 不能证明全球 kelp 监测、长期生态趋势、地下/水下 biomass 或实际治理效果
```

#### 发散方向

1. **遥感指数 baseline 路线**：用 NIR/SWIR、NDVI/NDWI 类指数、land/cloud mask 和简单阈值生成 kelp candidate。优点是可解释、低成本；风险是海面反光、云和近岸陆地会制造 false positives。
2. **轻量分割模型路线**：用 U-Net / DeepLab 类模型输入多通道 bands 和辅助层，直接输出 mask。优点是贴合 target_form；风险是训练集小、空间相关和标注噪声导致过拟合。
3. **两阶段路线**：先判断 tile 是否存在 kelp，再对 positive tile 分割。优点是控制空 tile false positives；风险是第一阶段漏检会直接丢失 mask。
4. **后处理路线**：用 land/cloud mask、morphological opening/closing、小连通域过滤和海岸线邻近约束清理输出。优点是降低明显错误；风险是把稀疏真实 kelp 删掉。
5. **标签质量路线**：按 cloud cover、mask 面积、边界不确定区域给训练样本分层或降权。优点是承认 noisy labels；风险是没有官方置信度时容易主观化。
6. **部署解释路线**：输出除了 mask，还要给生态管理者看 false positive 风险、不可观测区域、云遮挡和“只能解释 canopy”的边界。优点是真实效用更强；风险是竞赛指标不直接奖励。

#### 主链收束

> 在测试样本独立、地理坐标不可用、外部数据受限的边界下，先建立可解释的 spectral-index / mask baseline，再训练多通道语义分割模型，并用 positive/empty tile、cloud/land boundary、mask area 和 Dice 分层验证；最终输出必须被解释为“Landsat 可见 kelp canopy segmentation”，而不是完整生态健康判断。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 指标复现 | true mask + predicted mask | Dice score | 手算 TP/FP/FN 与官方指标对照 | 如果 empty mask 处理不清，先修指标 |
| baseline | bands + land/cloud/DEM | spectral threshold mask | Dice + positive recall + false positive rate | 如果全空或全海面预测得分看似高，必须拆分 positive tiles |
| 模型闭环 | 多通道 tile + train mask | segmentation mask | holdout / K-fold by tile，尽量按 mask area/cloud/land 分层 | 如果只随机像素切分，停止 |
| 边界验证 | land/cloud/invalid layers | masked / postprocessed output | land false positive、cloud false positive 检查 | 如果后处理删除真实 kelp，回退 |
| 独立性审计 | test inference code | per-tile independent prediction | 检查代码不读取其他 test images 生成伪标签 | 如果跨 test sample 学习，停止 |
| 代理结论审计 | mask + report | claim ladder | 每个生态结论追溯到可观测层 | 如果把 canopy mask 写成 biomass/health，降级表述 |
| 部署样张 | 1-3 张示例 tile | mask + uncertainty / invalid-area note | 人工检查可读性 | 如果生态管理者无法理解风险，不能算真实效用完成 |

### 10.3 查阅答案后的真实做法

答案来源包括 Kelp Wanted winners repo、winners blog、第一名 technical report、第一名 README、第二名 winner documentation、第三名 README/write-up 和 Top MATLAB User README。

公开答案的关键做法包括：

1. Winners repo 明确 labels 来自 Floating Forests citizen scientists，前三名 private Dice 非常接近：第一名 0.7332、第二名 0.7318、第三名 0.7296。
2. Winners blog 总结共性做法：spectral indices 很关键，尤其是 SWIR、NIR、Green 相关特征；赢家普遍使用 flips、rotations、cutmix、mosaic 等 augmentation；四个获奖方案都是 deep learning ensembles，常见架构包括 UNet、ConvNext、Swin。
3. 第一名 Team Epoch IV 用 NDVI、NDWI、ONIR 及 NDVI 派生的 edge detection、contrast enhancement 等特征；先训练 XGBoost，其预测再作为 deep learning 模型的新特征；最终 ensemble 包含 VGG-based UNet、SwinUNETR、ConvNext，并对 raw logits 加权平均后阈值化。
4. 第一名 technical report 说明把 NaN 值 `-32768` 置零，使用 StandardScaler、Gaussian blur target、weighted FocalDiceBoundary / Dice loss、8 种 flip/rotation TTA，并在 validation set 上优化阈值。
5. 第一名同时记录了许多未成功路线：移除少量疑似错误标签、stacked ensembling、Prithvi/TorchGeo、YOLO、非 Dice 类 loss、基于 band range 的裁剪、dithering、EfficientNet、额外亮度/对比度 TTA 等。
6. 第二名明确做了 EDA，发现 validation data 中有 duplicate / correlated AOIs。由于没有 CRS，他用 DEM layer 生成 ResNet embeddings，以 cosine similarity > 0.97 归为同一 AOI group，再做 10-fold stratified CV。
7. 第二名发现 corrupted / mislabeled images，并强调 submission preview 很重要：每次生成 submission 后自动画出样例预测，避免因为没有视觉检查浪费提交。
8. 第二名最终用 base spectral bands + 17 个 spectral indices，并通过 weighted sampler、mixed precision、threshold tuning、ensemble weights 提升表现；单纯 XGBoost per-pixel classifier 只到约 0.5125，不能替代深度分割模型。
9. 第三名 write-up 明确指出 citizen-science annotations 有大量噪声和误标，部分 image/mask 似乎存在 shift；如果真实部署，应先 curated updated dataset，再重新训练和调参。
10. 第三名最终只使用 SWIR、NIR、Green 三个通道训练，使用 altitude channel 生成 sea-land mask，在后处理中移除陆地预测，并用 0.43 阈值转成二值 mask。
11. Top MATLAB User 使用 19 个 U-Net CNN ensemble，说明本任务强依赖 segmentation ensemble 和训练资源，而不是单一轻量模型即可稳定解决。
12. 多个赢家报告了较高计算预算：第一名每个模型训练约 3-6 小时且使用 A5000/A6000/H100；第二名使用 RTX 3090、Azure T4/A100，最终 ensemble 训练每模型 2-7 小时；第三名训练约数天。

候选规则：

1. **标注来源与空间配准规则**：如果项目依赖人工标注、遥感标签、问卷答案、转录文本或历史记录，必须记录 annotation_source、noise_modes、alignment_or_registration、corrupted_or_mislabeled_samples 和 curation_rule。
2. **相关样本与重复单元规则**：即使题面移除了地点、用户、学校或时间信息，也要检查是否存在 correlated duplicates；若有合法 proxy，应做 group-aware validation，不能让近重复样本同时进入训练和验证。
3. **可观测代理边界规则**：凡任务测的是代理信号，必须写清 observed_signal、latent_construct、unobserved_parts、confounders_or_occlusions 和 claim_limit。
4. **输出抽样预览规则**：凡输出是图片、地图、mask、压缩包、表格、报告或提交文件，必须在正式提交/交付前抽样预览，检查文件结构、尺寸、命名、空输出、异常值和可读性。
5. **领域基线规则**：baseline 不能只是最简单模型，还必须包含领域上“人会先想到的解释性规则”，例如遥感指数、水文常年值、城市均值、简单阈值或手工规则。

题面事实：

1. Youth Mental Health Narratives 是 CDC/DrivenData 围绕 NVDRS youth suicide narratives 的双赛道挑战，目标是提高 youth suicide research 的数据质量、覆盖和分析能力。
2. NVDRS 数据来自 law enforcement reports、coroner/medical examiner reports、toxicology reports、death certificates 等来源，经 state-level abstractors 摘要和编码后进入 CDC 管理的数据库。
3. 数据样本为 2017-2021 年、5-24 岁 completed suicides 的 de-identified narratives，但仍包含高度敏感、机密的细节。
4. 题面明确提醒 narratives 可能令人不适，应优先照顾参与者自身心理健康，并给出 988 Suicide & Crisis Lifeline。
5. Automated Abstraction track 是监督式代码执行任务：用 `NarrativeLE` 和 `NarrativeCME` 预测 24 个 standard variables，包含二元变量和两个多分类变量。
6. AA 的目标是自动化标准变量抽取，服务 manual abstraction streamline 和 data quality control，不是对个人进行风险预测或临床诊断。
7. AA 评分是 multi-variable average F1：二元变量用 binary F1，多分类变量用 micro-averaged F1，再跨变量平均；tie-breaker 是 inference time。
8. AA 是 code execution challenge：提交 `submission.zip`，根目录必须有 `main.py`，读取 `data/test_features.csv`，生成同目录 `submission.csv`；官方建议先本地 Docker `make test-submission`，再 smoke test，再 full scoring。
9. Test narratives 有 1,000 条；测试时 logs 被关闭以保护数据机密，smoke test 只跑小样本且启用日志。
10. 外部数据和模型允许，但必须自由公开、开放许可；禁止把竞赛数据上传到会保留数据的第三方服务或 API，例如 ChatGPT/Gemini。可下载开源模型权重在本地或可清理数据的云环境运行。
11. 默认每个 test case 推理时独立处理，不得利用其他 test cases 的信息作为训练特征或标签，例如 test pseudo-labeling / unsupervised learning。
12. 参赛者不能分享、复制、发布竞赛数据；比赛结束后应删除 raw competition data。数据发布还受到小样本 cell suppression、不可组合识别、CDC 非背书和 disclaimer 等限制。
13. Novel Variables track 是无标签、开放式研究报告任务：从 narratives/free text 中提出尚未被 existing standard variables 捕获、但能推进 youth mental health research 的新变量。
14. NV 要求 submission 是 4 页以内 PDF，回答 key findings、文献/理论依据、方法、技术贡献、评价、代码片段、可视化等问题。
15. NV 由 NVDRS 和相关研究专家按 rubric 评审：Technical novelty 30%、Insight 25%、Rigor 25%、Communication 20%。题面鼓励使用 suicide theory、现有研究和 NVDRS Coding Manual，而不是只展示已有标准变量能做的分析。

#### 验证切分、场景异质性与目标形态

```yaml
validation_semantics: AA 为隐藏 test narratives 上的代码执行；NV 为专家评审的开放式研究报告；本地验证需模拟 variable-level F1、rare labels、LE/CME source differences 和报告 rubric
segment_or_context: age band, year, state/source abstractor style, LE vs CME narrative, mental-health variables, disclosure variables, incident variables, rare labels, categorical variables, free-text fields
target_form:
  automated_abstraction: 24 个 binary/categorical standard variables
  novel_variables: 文献支撑的新变量定义 + 抽取方法 + 研究价值报告
metric_structure:
  automated_abstraction: per-variable F1 平均；binary F1 与 categorical micro-F1 混合；tie-breaker inference time
  novel_variables: expert rubric = technical novelty 30 + insight 25 + rigor 25 + communication 20
ignored_or_valid_region: 隐私合规、cell suppression、禁止公开原文或可再识别片段；超过页数的报告内容不被评估
evidence_or_label_quality: standard variables 来自人工 abstraction；narratives 是摘要而非原始报告；不同来源和抽象者可能遗漏、解释不一或编码错误
correlated_or_duplicate_units: 同州/同年份/同 abstractor/source 风格可能相关；同一 case 的 LE/CME 是配对信息而非独立样本
case_context_contract:
  unit_of_prediction_or_action: one individual case / uid
  same_batch_visible_information: 当前 case 的 NarrativeLE 和 NarrativeCME
  cross_case_forbidden: test case 之间伪标签、统计适配、无监督 test 学习、由 test 分布调参
prediction_time_contract:
  actor: frozen inference script 或研究报告作者
  action_time: container inference / report submission before expert review
  visible_information: official train data、公开文献、coding manual、开源模型权重
  invisible_information: hidden test labels、其他 test case-derived features、未授权原始报告、赢家答案
observable_proxy_boundary:
  observed_signal: abstractor-written LE/CME narratives and coded variables
  latent_construct: youth suicide circumstances, mental health history, social context, prevention-relevant factors
  unobserved_parts: full original reports, lived experience, causal mechanisms, clinical diagnosis quality, counterfactual prevention effect
  confounders_or_occlusions: reporting bias, family/police/medical examiner source differences, abstraction instructions, stigma, missing detail
  claim_limit: 可用于研究变量抽取和质量控制，不能直接用于个人风险预测或因果政策断言
label_or_measurement_quality_contract:
  source_or_annotator: state-level abstractors following NVDRS coding manual
  measurement_or_annotation_process: LE/CME reports -> narrative summaries -> standard variable abstraction -> CDC QA/integration
  noise_modes: human error, inconsistent abstraction, missing source detail, ambiguous language, rare label undercoverage
  curation_or_exclusion_rule: sensitive data cannot be published; small cells suppressed; derived outputs must avoid reidentification
execution_contract:
  runtime_or_site: DrivenData code execution / local Docker / smoke test
  input_paths_or_materials: data/test_features.csv, data/submission_format.csv
  output_format: submission.csv or 4-page submission.pdf
  dependency_or_tooling: open-source licensed packages/models only
  network_or_permission: no retained third-party APIs; no data sharing
  time_memory_compute_limits: must fit container and inference-time tie-breaker
downstream_use: researchers, CDC/NVDRS data quality teams, policymakers; affected groups include youth and families represented in narratives
chosen_strategy: AA 先做 coding-manual/rule/keyword + local open-model baseline，再做 per-variable fine-tuning/thresholding；NV 先做 literature gap matrix + existing-variable coverage check，再做 reproducible extraction and expert-readable 4-page report
what_this_validation_can_prove: 模型/变量能否从 de-identified narratives 抽取研究有用信号
what_this_validation_cannot_prove: 不能证明个体风险、因果机制、政策有效性或所有州/年份/群体公平可用
```

#### 发散方向

1. **AA 规则/关键词 baseline 路线**：按 NVDRS coding manual 和变量定义建立可解释 baseline。优点是低泄漏、可审计；风险是召回不足、处理不了含蓄叙述。
2. **AA 本地开源 LLM/长文本模型路线**：使用 Longformer/BigBird/DeBERTa 类长文本模型或本地开源 LLM。优点是适配长 narratives；风险是高算力、隐私和许可证边界。
3. **AA per-variable 专门化路线**：每个变量单独建模、阈值校准、处理罕见标签和类别不均。优点是贴合 multi-variable F1；风险是过拟合和维护成本高。
4. **AA source-aware 路线**：分别处理 LE/CME narrative，再融合。优点是尊重数据生成机制；风险是某些变量只在一类来源中更常出现。
5. **NV 文献驱动路线**：从 suicide theory、surgeon general / national strategy、NVDRS 既有研究中找未覆盖变量。优点是更容易获得专家认可；风险是变量不可稳定抽取。
6. **NV 数据驱动路线**：用 topic modeling、embedding clustering、local LLM coding、manual codebook 发现常见未编码主题。优点是能发现意外模式；风险是 post-hoc story 和隐私泄漏。
7. **NV 研究转化路线**：每个新变量都写 definition、coding rule、examples-as-paraphrase、prevalence under suppression、validation method 和 how-it-advances-research。优点是落地；风险是 4 页空间极紧。

#### 主链收束

> 在高度敏感、不可上传第三方 API、测试样本独立和公开发布受限的边界下，把任务拆成两条主链：AA 以本地可复现的长文本/规则模型预测 24 个 standard variables，并按 variable-level F1、稀有标签、LE/CME 来源和容器执行验证；NV 以文献-变量缺口矩阵为起点，提出可定义、可抽取、未被现有变量覆盖的新变量，并用严格隐私保护和专家 rubric 写成 4 页研究报告。

#### 最小闭环

| 模块 | 输入 | 输出 | 验证方式 | 失败边界 |
| --- | --- | --- | --- | --- |
| 隐私与工具边界 | rules / DSA / model plan | data handling checklist | 检查无第三方 retained API、无原文外发、可删除 raw data | 如果需要把文本发给 ChatGPT/Gemini，停止 |
| 指标复现 | train labels + baseline predictions | multi-variable F1 | sklearn f1_score 对照官方脚本 | 如果只看总体 accuracy，停止 |
| AA baseline | train narratives + variable definitions | keyword/rule/local model predictions | per-variable F1 + rare label recall | 如果罕见变量全预测 0，不能进入复杂模型 |
| AA 执行契约 | main.py + dummy weights | submission.csv | runtime `make test-submission` + smoke test | 如果根目录/路径/列类型不对，停止 |
| AA 输出抽检 | submission.csv | schema and value report | 列顺序、dtype、范围、缺失、全零变量检查 | 如果输出格式错，先修格式 |

### 11.3 查阅答案后的真实做法

答案来源包括 youth-mental-health winners repo、winners blog、AA 前四名 README/报告、NV 前三名 README/最终报告和 write-up。

公开答案的关键做法包括：

1. Winners repo 显示 AA 第一名 private F1 0.8660，第二名 0.8650，第三名 0.8636，第四名 0.8624；NV 第一名提取 temporal information 构建 suicide 前事件时间线，第二名识别 narratives 中常见但未被标准变量跟踪的概念，第三名研究 non-normative online engagement。
2. Winners blog 总结，所有获奖方案主要依赖 general-text LLMs，而不是医疗/法律专用模型；AA 赛道赢家用 BigBird、Longformer、DeBERTa、Gemma、Llama、Qwen 等进行 fine-tuning / inference，三四名还用 Qwen、Mistral、Llama、Phi、Yi 进行数据生成与增强。
3. AA 第一名将任务视为 long-document multilabel classification，使用 BigBird + Longformer，BigBird 做 masked-language-model pretraining，使用 symmetric Lovász loss，5-fold ensemble，并为每个 binary variable 单独优化 threshold。
4. AA 第一名记录了强 baseline：TF-IDF + Logistic Regression / LightGBM 约 0.74-0.75，Mistral prompt engineering 约 0.72，Longformer/BigBird fine-tune 才显著提高；这说明复杂 LLM 前必须有传统文本 baseline。
5. AA 第一名训练成本很高：2 张 V100 约 10 天，推理约 1 小时；第二名也使用 H100，且记录了 local CV 与 public/private leaderboard 的不一致。
6. AA 第二名使用 prompt-based fine-tuning，把变量描述转成 `[MASK]` prompt，让模型预测 yes/no token；用 MultilabelStratifiedKFold、per-variable F1/micro-F1、threshold/ensemble weight optimization。
7. AA 三四名使用 LLM 生成额外数据、半监督/soft labeling 和 LoRA 训练，说明 synthetic / generated data 可以作为训练材料，但必须受数据安全、许可和可复现边界约束。
8. Winners blog 指出最难预测的是 disclosure to family/friend/intimate partner 等变量，因为模型需要理解语境：提到某人不等于 disclosed intent to that person；最容易的是 weapon type、anxiety、bipolar、ADHD 等关键词强变量。
9. NV 第一名以“victim’s story / timeline”为抓手，筛选 3201 条含时间信息 narratives，用 Flan-T5 prompts 抽取 temporal concepts、sentence topics 和 relative hours，形成 event log 和 Sankey diagram；人工标注 40 个样本验证，sentence extraction F1 0.97、topic accuracy 0.87、temporal extraction accuracy 0.88。
10. NV 第二名 HealthHackers 使用三条路径：topic modeling、LLM response clustering、combined pathway；两个 medical doctors 独立审查并达成共识，最后提出 45 个 novel variables，并把它们分成无法映射到 NVDRS manual 的新变量和可关联到已有类别的变量。
11. NV 第三名 UM-ATLAS 以 social media / online spaces 为理论框架，结合 Durkheim 和 IMV suicide theory、Surgeon General social media advisory 等文献；先用关键词和 log-odds/embedding 找 1,279 条高密度 narratives，再由 8 位作者做 thematic content analysis 和 codebook，手工标注 472 条并计算 IAA、precision/recall/F1。
12. NV 第三名证明“新变量”不是泛泛主题：它给出 Private Sharing、Conflict、Victimization、Withdrawal、Harmful Exposure 五个 code，分别有定义、理论依据、人工标注、IAA、LLM prompt 评估和误差分析。
13. Midpoint bonus 中 MPWARE 的 ReportsDiscrepancy variable 关注 LE/CME narratives 不一致，Jackson5 关注 panic attacks，说明专家会奖励能暴露数据质量或重要未编码临床/公共健康因素的变量。

| --- | --- | --- |
| 敏感数据边界是硬约束 | 赢家使用本地/可控环境运行开源模型，三四名生成合成数据但仍在可复现代码内 | 协议需要 sensitive_data_handling_contract：不能只写“可用 AI/不可用 AI”，要写数据保留、运行位置、许可、删除和发布规则 |
| 需要规则/关键词 baseline | AA 第一名展示 TF-IDF/LightGBM/logistic 与 prompt LLM baseline；复杂模型提高但基线很强 | baseline-first 继续强化，且要包含传统强基线而非只上 LLM |
| 多变量 F1 需要分变量验证 | 赢家做 per-variable threshold、per-variable F1；blog 指出 disclosure 变量特别依赖语境 | metric_structure 需要 variable-level error analysis；平均分会掩盖最需要语义理解的变量 |
| 长文本需要专门处理 | 赢家使用 Longformer/BigBird/DeBERTa prompt-based fine-tuning、global attention、collate 截断优化 | execution_contract 要记录 context window、truncation/padding、长文本丢失信息风险 |
| 资源预算要前置 | 第一名 2 V100 10 天，第二名 H100，NV 第一名 A100 运行 Flan-T5-xl | reproduction_budget 在 AI 项目里尤其要写“低成本代理 + 何时值得上大模型” |
| 变量发现要有专家/文献支撑 | NV 一等奖用 timeline story + 文献；二等奖双医生独立 review；三等奖理论框架 + codebook + IAA | 协议需要 novel_variable_or_construct_contract 和 expert_review_contract |
| 新变量不是主题词 | NV 三等奖给出变量定义、codebook、人工标注、IAA、模型 precision/recall/F1 和误差分析 | 公共议题/变量提取必须满足 definition、non-duplication、extractability、validity、action/research value |
| 代理目标不能夸大 | Narratives 是 abstracted summaries；变量支持研究和数据质量控制，不等于个体风险预测或因果政策结论 | observable_proxy_boundary 继续被验证 |
| 人在读材料时也有风险 | 题面给出 988 Lifeline 并提醒材料可能令人不适 | 协议需要 operator_wellbeing_or_safety：处理创伤、心理健康、暴力、敏感个案时要保护执行者 |
| 沟通产物是评分核心 | NV 最终是 4 页 PDF，赢家用 Sankey、变量清单、文献引用、方法图和专家可读叙事 | communication_output_contract 继续强化：报告不是装饰，而是专家评审的主交付物 |
| 数据质量本身可成为公共议题 | MPWARE 的 LE/CME discrepancy 获 midpoint bonus；NV 三等奖发现 social media disclosure 标准变量漏记 | 协议应允许把“材料之间的不一致/缺失/不可见”提升为问题本身，而非只当清洗噪声 |

### 11.5 写回协议的候选规则

当前证据等级：

| 规则 | 证据等级 | 理由 |
| --- | --- | --- |
| 正式指标 / 真实效用双指标 | E2 | P10 中 F1、专家 rubric 与真实 suicide research utility 明显不同 |
| 资源边界显式化 | E2 | P10 强化敏感数据、第三方 API、模型许可、数据删除和发布限制 |
| baseline-first 闭环 | E2 | P10 的 TF-IDF/LightGBM/logistic/keyword baseline 证明强基线必要 |
| 指标结构拆解 | E2 | P10 的 multi-variable F1、categorical micro-F1、variable-level difficulty 继续支持 |
| prediction_time_contract / case_context_contract | E2 | P10 明确 test case 独立，且模型必须自动运行在 hidden narratives 上 |
| execution_contract / reproduction_budget | E2 | P10 代码执行、长文本推理、GPU/时间成本和 inference-time tie-breaker 强支持 |
| downstream_use / communication_output_contract | E2 | P10 NV 赛道完全由专家报告评分，且服务公共健康研究 |
| observable_proxy_boundary | E2 | P09/P10 均强支持：可见 canopy / abstracted narratives 不能直接推出潜在生态或个体心理事实 |
| label_or_measurement_quality_contract | E2 | P09/P10 均强支持：citizen science labels 与 NVDRS abstracted variables 都有来源、噪声和配准/抽象过程 |
| output_sanity_preview | E2 | P09 提交预览，P10 submission.csv/schema/PDF 页数和敏感发布都需要抽检 |
| sensitive_data_handling_contract | E1 | P10 强支持：敏感文本、禁止 retained API、删除 raw data、suppression 和 disclaimer |
| operator_wellbeing_or_safety | E1 | P10 强支持：题面明确心理健康风险与 988 Lifeline |
| expert_review_contract | E1 | P10 强支持：Novel Variables 赢家依赖医生、公共健康专家、IAA 或人工评审 |
| novel_variable_or_construct_contract | E1 | P10 强支持：新变量需要理论依据、定义、可抽取性、非重复性和研究价值 |
| mixed_methods_validation | E1 | P10 强支持：机器发现 + 人工 codebook + IAA + expert consensus |

候选规则：

1. **敏感数据处理规则**：如果材料涉及心理健康、医疗、未成年人、暴力、隐私、学校或可再识别个案，必须记录 data_sensitivity、allowed_environment、prohibited_transfer、retention/deletion、publication_suppression 和 disclaimer。
2. **执行者安全规则**：如果材料可能造成心理负担或替代性创伤，Plan 必须写明 exposure_limit、support_resource、opt-out、轮换和复盘机制。
3. **专家复核规则**：公共健康、教育、医学、心理、法律、政策等项目不能只用模型自评。必须定义 expert_role、review_sample、adjudication_or_consensus、IAA/一致性或专家意见如何进入决策。
4. **新变量/公共议题契约规则**：一个新变量或公共议题必须同时满足 definition、gap_from_existing_variables、theory_or_literature_basis、extractability、validation_plan、action_or_research_value 和 communication_form。
5. **生成数据状态规则**：LLM 生成、软标签、模拟访谈、synthetic case 和故事样例必须明确 synthetic/training-only/report-example 状态；不能把生成材料写成真实观测。
6. **材料不一致升格规则**：当两个材料源、叙述或群体之间不一致时，先判断它是否是噪声、偏差、缺失机制，还是值得研究的公共问题本身。
