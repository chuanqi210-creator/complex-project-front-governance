# Complex 项目持续治理：低摩擦用户入口

current_as_of：2026-06-22

用途：把厚协议压缩成用户能自然使用的入口。用户不需要知道 trace 名称，也不需要填写表格；AI 负责识别最终主张、恢复当前状态、构造新领域治理镜头、查真实项目和最佳实践、调用经验库、选择动态路由、推进 Loop 小闭环、写入治理文件并压缩反馈。

## 一、5 行启动入口

用户只需要这样说：

```text
这是一个重大项目。
目标是：……
已有材料在：……
我希望结果达到：……
请按 Complex 项目持续治理协议推进，但不要把流程负担转嫁给我。
```

如果用户没有写全，AI 也应自动补齐：

1. 从对话和文件中恢复目标。
2. 识别最终最想证明的主张。
3. 选择一个主类和必要的高风险标签。
4. 启用 `new_domain_governance_builder`，生成本轮临时治理镜头。
5. 从经验库索引抽取 1-3 个相近失败模式校准。
6. 只问最多 1-2 个会显著降低返工的问题。

## 三、大类识别

AI 不要求用户选择具体 trace，只在内部识别：

| 主类 | 常见项目 |
| --- | --- |
| `research_data_model` | 研究、实验、数据、模型、调研、评价、算法治理 |
| `engineering_software_product` | 工程、软件、产品、开源、原型、企业交付 |
| `education_practice_public_service` | 教育、社会实践、公益、公共服务、志愿服务 |
| `organization_business_governance` | 组织、商业、政策、程序、公平、验收、资源配置 |
| `communication_culture_design` | 传播、文化、设计、科普、舆情、国际交流 |
| `high_risk_regulatory` | 横切标签：健康、安全、法律、隐私、未成年人、财务、食品、交通、心理、知识产权等 |

如果一个项目同时属于多个类别，优先围绕“最终要证明的主张”选择主类，高风险标签必须额外保留。

## 四、AI 对用户的默认回复形态

用户可见回复控制在 5 项以内：

1. 我识别这个项目最终想证明什么。
2. 我会按哪一类主路由推进，以及是否触发高风险标签。
3. 我会启用哪些通用镜头。
4. 当前最大不能外推风险是什么。
5. 下一步我会产出什么文件或动作。

当用户明确只要人看版时，默认不展示机器看版、长 YAML、验证器字段或内部线程账本；只用普通语言说明本轮判断、改了什么、还有什么风险。机器状态写入治理文件，不转嫁给用户阅读。

不向用户展示：

1. 完整 YAML 模板。
2. 全部经验库条目。
3. 全部失败模式清单。
4. 全部执行闸门。
5. 长篇机器看版。
6. 子代理内部任务分解细节，除非它影响用户决策或需要人工代理执行。

这些内容写入治理文件。

## 五、低摩擦反馈闸门

只有在会显著降低返工时才问用户，优先问：

1. 这个项目最终最想证明什么？
2. 是否存在健康、安全、法律、隐私、未成年人、财务、食品、交通、心理、知识产权等高风险边界？

如果用户已经给出足够语境，AI 直接记录 assumption 并继续。

## 六、反回退规则

1. 用户没有说“持续治理协议”，但任务满足重大项目触发条件，也可以触发协议。
2. 用户说“继续”“继续任务”“继续优化”，AI 必须恢复最新机器看版，而不是重开。
3. 用户临时加想法，视为隐性评价函数暴露，不视为打断。
4. 用户要求“真实项目”，不得只看当前目录、目录名称、当前本地 AI 科研材料或上一轮主题，也不得反向排除外部 AI/理工真实项目。
5. 用户要求“新领域”，不得先问用户要哪个 trace；AI 应先构造临时治理镜头。
6. 用户要求“不要停止”，每轮必须有写回、扫描和 next_route。

## 七、最小成功标准

低摩擦入口成功，不是用户少说话，而是：

1. 用户不用学习协议术语。
2. AI 不漏掉重大项目治理动作。
3. 用户能轻松纠偏。
4. 文件和机器看版能恢复。
5. 最终成果更具体、更可追踪、更能落地。
6. 主协议不会因为新领域不断变成场景目录。
