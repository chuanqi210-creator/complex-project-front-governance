# 前置治理协议优化任务：十阶段治理记录

本文不是新的提示词堆叠，而是一次“用前置治理协议优化前置治理协议”的过程记录。核心目标是把 auto-research 中有价值的结构思想迁移进原 Word 协议，使它从“阶段提示词链”升级为“复杂项目启动前的状态治理系统”。

## 阶段 0：总控启动

### 人看版

本次任务的对象不是普通科研项目，而是一个“元协议优化项目”：优化你写的《复杂项目启动前置治理协议》。因此 AI 不能只做 Word 美化，也不能只把提示词改长，而要先理解协议想解决的问题。

原协议的初衷是降低 AI 分析上下文的摩擦，让 AI 更好地理解前置问题，并主动利用材料、skill、工具和外部方法。auto-research 的价值在于，它不是靠单次 prompt 维持上下文，而是用文件化状态、阶段门、材料登记、决策记录和路由规则来让研究过程可恢复。

### 机器看版 handoff

```yaml
current_stage: stage_0_control_start
task_type: meta_protocol_optimization
object_to_optimize: 复杂项目启动前置治理协议.docx
core_intent:
  - 降低复杂项目前置沟通摩擦
  - 让 AI 在不跑偏的前提下主动发挥能力
  - 把材料、skill、外部方法、阶段判断接入前置治理
main_constraint:
  - 不把任务变成 Word 排版项目
  - 不把优化理解为单纯提示词润色
next_route: stage_1_context_and_intake
```

## 阶段 1：项目情景识别与材料 Intake

### 人看版

这个项目属于“AI 协作方法论 / 项目治理协议 / 复杂系统启动流程设计”。它不是科研论文、工程代码或产品方案本身，而是服务这些复杂任务的上层工作流。

当前最大不确定性不是“Word 好不好看”，而是：如何让协议既保持阶段纪律，又保留 AI 主动发现、主动搜索、主动组织问题的能力。

材料 intake 判断如下：

| 材料 | 角色 | 权威性 | 是否进入当前依据 |
| --- | --- | --- | --- |
| 原 Word：复杂项目启动前置治理协议.docx | 原始协议主体 | authoritative | 是 |
| auto-research README / AGENTS / templates | 结构思想来源 | authoritative for repo design | 是 |
| 之前稀释法 pilot | 真实项目验证案例 | provisional | 部分进入 |
| teacher feedback | 外部审阅意见 | authoritative user context | 是 |
| Word 渲染图/PDF | 交付检查材料 | output | 否，不作为思想依据 |

本阶段最不该急着做的是：直接改成一个更长、更复杂的 prompt 模板。因为这会让协议看起来更完整，但没有解决上下文继承和阶段恢复问题。

### 机器看版 handoff

```yaml

sub_types:
  - research_workflow_governance
  - project_startup_context_engineering

  - 原 Word 的 0-10 阶段结构
  - auto-research 的 intake/state/question/decisions/closure-routing
  - 用户初衷：降低前置问题表达摩擦，释放 AI 能力
not_current_basis:
  - Word 渲染图
  - 纯格式美化
  - 之前生成的教师展示 PDF
largest_uncertainty: 如何在 AI 自由发挥和阶段治理之间建立可操作平衡
next_route: stage_2_divergence
```

## 阶段 2：高质量发散

### 人看版

基于原协议和 auto-research，可以发散出 8 个优化方向：

| 方向 | 核心问题 | 价值 | 风险 |
| --- | --- | --- | --- |
| A. 提示词链增强 | 把每阶段 prompt 写得更清楚 | 最容易落地 | 只会变长，不一定更可恢复 |
| B. 材料 intake 层 | AI 先登记材料身份和权威性 | 防止旧稿、草稿、生成物混入当前依据 | 会增加启动步骤 |
| C. state 恢复入口 | 后续 AI 先读当前状态 | 降低多轮上下文摩擦 | 如果写太长会变成流水账 |
| D. question contract | 区分世界问题、经验规格、操作问题 | 特别适合科研/工程复杂问题 | 对简单任务略重 |
| E. decisions 记录 | 记录关闭、回退、阻塞、范围变化 | 方便老师/后续 AI 复盘 | 需要用户接受记录纪律 |
| F. closure routing | 用未关闭不确定性决定下一阶段 | 防止直接从材料跳到论文/代码/展示 | 需要清楚阶段门 |
| G. skill 搜索协议 | 搜索工具前先说明阶段缺口 | 能更好利用 AI 外部能力 | 可能过度搜索 |
| H. 最小闭环验证 | 先做最低成本验证 | 防止过早工程化/写作 | 需要接受不一次做完 |

判断：A 只能作为表层优化；B-F 是核心结构优化；G-H 是能力释放和执行控制。

## 阶段 3：Skill / 工具 / 外部方法搜索

### 人看版

本阶段不需要盲目安装新工具。最值得借鉴的是 auto-research 的结构设计，而不是某个具体插件。

从 auto-research 迁移的关键方法：

| auto-research 思想 | 原本作用 | 迁移到前置协议中的形式 |
| --- | --- | --- |
| `intake.md` | 登记用户材料 | 阶段 1 增加材料 intake，不再直接把材料丢进上下文 |
| `state.md` | 项目恢复入口 | 每阶段机器 handoff 必须能恢复 current_stage、current_basis、next_action |
| `question.md` | 问题契约 | 阶段 4 前后区分世界问题、经验规格、操作问题 |
| `decisions.md` | 决策日志 | 阶段关闭、回退、阻塞、用户选择都要记录 |
| `closure-routing.md` | 阶段路由 | 阶段 4 和阶段 9 判断下一步该走哪里 |
| evidence hygiene | 证据卫生 | 评分体系区分 idea、synthetic、literature、field、human-reviewed |
| skills optional | skill 可用但不强制 | 阶段 3 搜索 skill，但必须服务阶段缺口 |

## 阶段 4：主链收束

### 人看版

一句话主链：

> 把原本依赖连续对话推进的“前置治理提示词链”，升级为一个能登记材料、恢复状态、区分问题层级、记录决策、路由阶段并保留 AI 主动性的复杂项目启动协议。

通俗版：

> 原版像“让 AI 按步骤问我”；优化版像“给 AI 一个项目启动驾驶舱，让它知道材料从哪来、现在依据是什么、下一步为什么做”。

专业版：

> 该优化将前置治理协议从 prompt-sequence 转化为 file/state-oriented project governance protocol，通过 intake、state recovery、question contract、decision log 与 closure routing 降低复杂任务上下文迁移成本。

保留进主链：

- 材料 intake；
- state 恢复；
- question contract；
- decisions；
- closure routing；
- skill 搜索服务阶段缺口；
- 评分和最小闭环。

进入 backlog：

- Word 美化；
- UI 化；
- 自动生成完整项目仓库；
- 数据库化状态管理；
- 针对某一领域写死模板。

### 机器看版 handoff

```yaml

  将前置治理协议从阶段提示词链升级为可恢复、可路由、可执行的
  AI 协作启动治理系统。
kept_scope:
  - intake
  - state
  - question_contract
  - decisions
  - closure_routing
  - scorecard
  - minimum_loop

  - UI
  - database

```

## 阶段 5：Goal 模式文本

### 人看版

这个优化任务的 Goal 应该是自然语言，而不是字段堆砌。Codex 的 Goal 模式可以接收自然语言，关键是目标要稳定、边界要清楚、成功条件要能验证。

### 可复制 Goal 文本

```text
优化《复杂项目启动前置治理协议》，使它从单纯的阶段提示词链升级为一个可恢复、可路由、可执行的复杂项目启动治理系统。

第一性原理：
1. 前置治理的目标不是限制 AI，而是降低上下文摩擦，让 AI 在清楚材料、状态、边界和阶段门的前提下主动发挥能力。
2. 任何复杂项目启动前，必须先弄清楚材料从哪里来、哪些材料是当前依据、当前最大不确定性是什么、下一步属于哪个阶段。
3. 人看版负责解释判断和取舍，机器看版负责让后续 AI 恢复状态。
4. 搜索、skill、工具和外部方法都必须服务某个阶段缺口，而不是为了显得高级。
5. 不把草稿、旧文件、生成输出、候选材料自动当成当前依据。

核心主链：
材料 intake -> 情景识别 -> 高质量发散 -> skill/工具/方法搜索 -> 主链收束与阶段路由 -> Goal -> Plan -> 评分体系 -> 最小闭环 -> 执行前闸门 -> 执行。

边界：
- 不优先做 Word 美化、UI、数据库化或自动化系统；
- 不把协议变成僵硬模板；
- 不为追求完整而牺牲可用性；
- 不把 AI 的自由发挥变成无依据发散。

成功标准：
优化后的协议必须让一个新 AI 能够知道：已有材料是什么、当前依据是什么、问题属于哪个阶段、下一步为什么这样走、什么时候应该停止或回退。
```

### 机器看版 handoff

```yaml

goal_status: stable
goal_type: natural_language_goal
invariants:
  - material_authority_before_analysis
  - state_recovery_before_continuation
  - route_before_execution
  - evidence_level_before_claim
  - human_output_and_machine_handoff
next_route: stage_6_plan
```

## 阶段 6：Plan 模式

### 人看版

当前阶段属于“架构期 + 收束期”。不是从零写新协议，而是在原协议骨架上做结构增强。

| 任务 | 为什么做 | 输出 | 验收 |
| --- | --- | --- | --- |
| 1. 保留原 0-10 阶段 | 原协议结构已经清楚 | 阶段链不被推翻 | 用户能认出原流程 |
| 2. 增加材料 intake | 解决上下文混乱源头 | 阶段 1 intake 表 | 能区分草稿/证据/输出 |
| 3. 增加 state recovery | 解决多轮继承问题 | 机器 handoff 字段 | 后续 AI 可恢复 |
| 4. 增加 question contract | 防止问题层级混乱 | 世界问题/操作问题区分 | 科研类项目可用 |
| 5. 增加 closure routing | 防止跳阶段 | 阶段 4/9 路由判断 | 能说明下一步为什么走 |
| 6. 重写 skill 搜索阶段 | 让工具服务缺口 | 工具矩阵 | 不盲装、不堆工具 |
| 7. 重写评分和最小闭环 | 防止虚假成熟 | scorecard + loop | 能决定继续/暂停/回退 |
| 8. 生成短入口 prompt | 降低使用门槛 | 最短总控提示词 | 用户一句话可启动 |

## 阶段 7：评分体系与终止条件

### 人看版

用这套标准判断协议优化是否成功：

| 指标 | 权重 | 0 分 | 1 分 |
| --- | --- | --- | --- |
| 保留原协议骨架 | 0.10 | 原 0-10 被推翻 | 原结构保留且增强 |
| 材料治理能力 | 0.15 | 仍把材料当普通上下文 | 能登记材料角色和权威性 |
| 状态恢复能力 | 0.15 | 后续 AI 仍要重读全部历史 | state/handoff 可恢复当前状态 |
| 阶段路由能力 | 0.15 | 固定顺序或凭感觉推进 | 能按未关闭不确定性路由 |
| AI 主动性 | 0.10 | 把 AI 管成填表机器 | 允许拓展但标注依据 |
| skill 搜索质量 | 0.10 | 工具堆叠 | 工具服务阶段缺口 |
| 最小闭环能力 | 0.10 | 直接大执行 | 先验证关键不确定性 |
| 可用性 | 0.15 | 太重，用户不愿用 | 有短入口和完整模式两种用法 |

终止条件：

- 如果只是把 prompt 写长，没有 intake/state/routing，不能结束；
- 如果协议复杂到无法用一句话启动，不能结束；
- 如果没有说明 auto-research 思想迁移到了哪里，不能结束；
- 如果无法用于一个真实技术卡点，不能结束。

## 阶段 8：最小闭环

### 人看版

第一轮最小闭环不需要做完整系统，只需要验证一个问题：

> 用户只给一个复杂想法或技术卡点时，优化后的协议能不能让 AI 先登记材料、恢复/形成状态、判断阶段，而不是直接写成品？

最小闭环设计：

| 模块 | 输入 | 输出 | 验证方式 |
| --- | --- | --- | --- |
| 短入口 prompt | 用户一句话目标 | AI 不直接执行，进入治理 | 看是否先问材料/状态 |
| intake | 文件/材料清单 | 材料 registry | 看是否区分权威材料和草稿 |
| state | 当前依据 | current_basis + open_questions | 后续 AI 能否接手 |
| routing | 未关闭问题 | next_route | 是否避免跳到成品 |
| scorecard | 当前成熟度 | 继续/回退/暂停 | 是否服务决策 |

以“技术卡点”使用时，推荐入口：

```text
完成研究：我现在的技术卡点是【写清楚卡点】。
已有材料包括：【文件路径、论文、实验记录、数据、草稿、截图】。
请先不要写论文、不要写最终方案。
请按前置治理协议先做 material intake、current_basis 判断、阶段路由和最小验证方案。
```

### 机器看版 handoff

```yaml
current_stage: stage_8_minimum_loop
minimum_loop:
  trigger: one_sentence_project_or_technical_blocker
  path:
    - intake
    - state
    - route
    - score
    - minimum_validation
proves:
  - protocol_reduces_context_friction
  - protocol_prevents_premature_execution
  - protocol_allows_ai_to_expand_with_evidence_labels
does_not_prove:
  - final_research_result_quality
  - all_domain_templates_are_complete
  - UI_or_automation_is_needed
next_route: stage_9_execution_gate
```

## 阶段 10：执行启动

### 人看版

实际执行的优化应该落在这些地方：

1. 在总控启动中加入“材料登记、状态恢复、阶段路由、人/机器双输出”的工作方式；
2. 将阶段 1 从单纯“项目情景识别”升级为“项目情景识别 + Material Intake”；
3. 将阶段 3 从“找工具”升级为“先判断能力缺口，再搜索 skill/工具/外部方法”；
4. 将阶段 4 从“主链收束”升级为“主链收束 + closure routing”；
5. 将阶段 5 的 Goal 明确为自然语言长期使命，而不是结构字段堆叠；
6. 将阶段 6 的 Plan 明确限定为当前阶段，不替代 Goal；
7. 将阶段 7 的评分体系加入证据等级和回退条件；
8. 将阶段 8 的最小闭环区分“方案可行性闭环”和“真实结果闭环”；
9. 将阶段 9 的执行前闸门加入 current_basis、旧稿隔离、route-back 检查；
10. 将阶段 10 的执行规则加入 state/decisions 更新要求。

这就是 auto-research 思想迁移后的核心变化：

> 原协议强调“按顺序问”；优化版强调“按状态推进”。  
> 原协议依赖“对话连续性”；优化版依赖“可恢复的机器状态”。  
> 原协议让 AI “主动拓展”；优化版让 AI “在证据、路由和主链约束下主动拓展”。
